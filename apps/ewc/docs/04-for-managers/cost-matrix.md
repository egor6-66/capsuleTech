# 💰 Матрица стоимости

> 🏠 [Хаб документации](../README.md) › 📊 [Для менеджеров](README.md) › **Матрица стоимости**

> **Аудитория:** 📊 Менеджер · 🛠️ Тимлид
> **Статус:** 🟡 Ориентировочные оценки

Связь «что мешает → какие возможности это блокирует → сколько примерно займёт». Все сроки — **ориентир** (±50%), не обязательство. Технические первоисточники — [Известные проблемы](../03-roadmap/known-issues.md) и [Технический долг](../03-roadmap/tech-debt.md).

---

## Препятствие → блокирует → срок

| 🚧 Препятствие | Что это блокирует для бизнеса | Кто чинит | Ориентир |
|---|---|---|---|
| [Виртуальный скролл таблицы](../03-roadmap/known-issues.md#виртуальный-скролл-таблицы) | Плавную работу на тысячах реальных строк | 🏗️ Платформа | Придёт с обновлением платформы — app-ресурс не тратит |
| [Анимация переходов](../03-roadmap/known-issues.md#анимация-перехода-между-страницами) | «Дорогое» ощущение на демо | 🏗️ + app | ~½ недели после готового решения |
| [Типизация](../03-roadmap/tech-debt.md#типизация) | Защиту от регрессий при росте кода | app + платформа | ~½–1 неделя |
| [Моки вместо API](../03-roadmap/tech-debt.md#временные-решения) | Работу на реальных происшествиях | app + 🔗 бэкенд | ~½ недели при готовом API |
| [Нет защиты входа](../03-roadmap/tech-debt.md#временные-решения) | Продакшн-безопасность | app + 🔗 бэкенд | ~½ недели |
| [Карточка только для чтения](../03-roadmap/backlog/platform.md) | Внесение и сохранение данных | app + платформа | ~1 неделя |

> [!IMPORTANT]
> Строки с 🏗️ закрывает **команда платформы**, а не команда EWC. Для плана приложения это значит: их можно вести **параллельно**, они не занимают разработчика EWC. Строки с 🔗 нельзя начать, пока заказчик не предоставит бэкенд-API.

## Сводка по фазам

Укрупнённо, в неделях для одного разработчика. Детализация в днях — в [бэклоге](../03-roadmap/backlog/README.md).

| Фаза | Что бизнес получает | Объём app-части | Зависит от |
|---|---|:---:|---|
| **Phase 1 — Стабилизация** | Надёжность, отсутствие регрессий, «дорогое» демо | ~1–2 недели | 🏗️ параллельная работа платформы |
| **Phase 2 — Реальные данные** | Боевая эксплуатация: настоящие данные, вход, сохранение | ~2–3 недели | 🔗 готовность бэкенда |
| **Phase 3 — Расширение** | Конкурентные фичи карты/таблицы, полировка UX | от 2 недель, инкрементально | приоритеты заказчика |

```
Сейчас        Phase 1         Phase 2          Phase 3
Demo    ──→   Стабилизация ──→ Реальные   ──→  Расширение
(готово)      ~1–2 нед         данные          (по выбору
                               ~2–3 нед 🔗      из бэклога)
```

> [!NOTE]
> **Phase 3 не имеет «общего срока» специально.** Это меню из [бэклога](../03-roadmap/backlog/README.md) (~35–50 дней идей суммарно). Берётся не целиком, а по приоритету заказчика — отсюда «инкрементально», а не одна дата.

## Как читать сроки

| Принцип | Что значит |
|---|---|
| 👤 Один разработчик | Оценки для одного человека, знакомого с платформой. Команда — быстрее, но не линейно. |
| 📏 ±50% | Это порядок величины, а не дедлайн. Уточняется при старте фазы. |
| 🏗️ Параллельно | Пункты платформы не тратят ресурс команды приложения. |
| 🔗 Зависит от бэкенда | Срок верен только при готовом API. Без него Phase 2 не стартует. |

## Что показать заказчику в первую очередь

Рекомендация по приоритету (максимум ценности за минимум усилий):

1. **Демо как есть** — 0 дней, всё [готовое](capabilities.md#готово-к-показу) работает.
2. **Phase 1** — снять «оговорки» и убрать риск регрессий перед ростом команды.
3. **Phase 2** — как только заказчик даст бэкенд: реальные данные сразу превращают демо в инструмент.
4. **Phase 3** — точечно под запрос: то, что заказчик назовёт важным на демо.
