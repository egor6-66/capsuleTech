# 📐 Контракт-система (`web-contract`)

> **Навигация:** 📍 [Обзор](README.md) · 🗺️ [Роадмап](roadmap.md) · 🏛️ [Архитектура](architecture.md) · 📐 [Контракты](contracts.md) · 🧰 [Creator](creator.md) · 📡 [Мониторинг](monitoring-flow.md) · 🏗️ [Платформа](platform.md) · ⌨️ [Web-code](web-code.md) · 🎁 [Free-wins](free-wins.md)

«Договор» компонента с редакторами. Компонент **сам** описывает, что он умеет и как с ним обращаться — редакторы/тесты/доки лишь **читают** это.

---

## 🧠 Ментальная модель

> 💡 **Контракт лежит В компоненте.** Button знает свой контракт, Table — свой. Редактор — не источник правды, а потребитель. Один контракт — много витрин: demo-стенд, ui/style-редакторы, тесты, доки.

```
@capsuletech/web-contract  (leaf, zero-dep)   → протокол: defineContract() + rule-примитивы
@capsuletech/web-ui  Button КАРМАНИТ свой     → ButtonContract = defineContract([...rules])
@capsuletech/web-table  несёт base+data+test
редакторы / demo / тесты  ПОТРЕБЛЯЮТ          → collectContracts()
```

🍃 **Почему отдельный leaf:** web-ui деклариует контракты, но web-ui **ниже** web-creator → тип контракта нельзя класть в creator (цикл). Leaf — общий язык продьюсера (web-ui) и потребителя (редакторы) без цикла. Это и есть **базовый контракт, без которого кит невалиден**.

## 🧱 Контракт = композиция мелких правил

Контракт собирается из переиспользуемых **rule-примитивов** + своего:

```ts
// web-contract — примитивы
rule.isLeaf()
rule.accepts(types[])          // структурное, parent-side
rule.props(zodSchema)
rule.variants([...])
rule.styleSlots([...])
rule.data(shape)               // data-bindable
rule.events([...])             // traceable
rule.recommend(pred, hint)     // мягкое

// web-ui/Button — композиция + своё
export const ButtonContract = defineContract([
  rule.isLeaf(),
  rule.props(z.object({ variant: z.enum([...]) })),
  rule.variants(['default','destructive','outline',...]),
  rule.styleSlots(['root']),
  rule.recommend(ctx => !ctx.hasLabel, 'Кнопке желателен лейбл'),
]);
```

Правила живут в тематических пакетах: `web-contract` — базовые примитивы, `web-table` — data-правила, `web-auth` — api-правила. Компонент берёт нужные + докидывает свои.

> 🔌 **Правила = «плагины».** Чем больше правил несёт компонент — тем больше правил/валидаций включает редактор. Инструменты **зажигаются по набору контрактов**: Table несёт `data` → появляется загрузчик/коэрция; Auth несёт `api` → появляется OpenAPI-панель.

---

## ⚖️ Правило №1 — две жёсткости

| Жёсткость | Поведение редактора |
|---|---|
| 🔴 **Constraint** | **блокирует** (нельзя дропнуть контейнер в Button; невалидный проп = ошибка) |
| 🟡 **Recommendation** | **подсказывает/варнит**, но разрешает (как `warn`-режим compliance) |

---

## 🧩 Правило №2 — два типа правил

- **Реляционные** (компонент ↔ компонент): вложенность, соседи, кардинальность. Пример: «Button не принимает контейнер в children».
- **Собственные** (своя поверхность): props-схема, варианты, styleSlots, data-shape, events, test-cases.

Оба композятся в один контракт.

---

## 🎯 Правило №3 — вложенность из одного источника

«Может ли A содержать B» объявляет **родитель** через `accepts` (single source). Ребёнок объявляет только **про себя** (`canBeRoot`, `requiresParent`). Так нет двойной декларации одной связи → чистая валидация.

---

## 🆚 Contract vs Compliance — две разные оси

`@capsuletech/compliance` уже есть и делает **родственное, но другое**. Не дублируем.

| | compliance (есть) | web-contract (новое) |
|---|---|---|
| Когда | build/dev, **AST исходника** | design-time/runtime, **JSON-дерево инстансов** |
| Субъект | import-граф между **слоями** | **компонент**: props/variants/data/events + accepts-children |
| Правило | «файл слоя X может импортить Y» | «инстанс A может содержать B», «какой проп» |
| Аудитория | разработчик (как пишет код) | конечный юзер (как собирает UI) + тулзы |
| Рантайм | build-only (Node/AST, тяжёлый) | browser-leaf (zero-dep) |

- **Архитектурные правила** (направление импортов, границы слоёв, no-fetch-в-Controller) → **остаются в compliance**. В контракт не тащим.
- **Компонентные правила** (capabilities, props, вложенность) → **web-contract**.
- 🌉 **Мост (на будущее):** реляционное правило вложенности объявляется один раз в контракте; редактор enforce'ит design-time, а compliance позже может **прочитать тот же контракт** и варнить на руками-написанный JSX. web-contract — zero-dep leaf, Node его импортнёт.

---

## ⚙️ Общее ядро правил — намерение, НЕ текущий шаг

У compliance и contract общий тонкий **rule-kernel**: тип правила `{ id, severity, message, check(ctx) }` + registry/extend + severity + форма результата + runner + formatter. Расходятся только **контекст и адаптер** (AST[Node] vs eval-дерева[browser]) — их слить нельзя.

**Целевое:** чистый leaf `@capsuletech/rules` (kernel), `compliance` и `contract` = два профиля над ним, единый формат violation (показывается одинаково в редакторе / build-логе / тесте).

> ⚠️ **Дисциплина:** сейчас НЕ объединяем. compliance работает — не трогаем. Контракт строим конкретно на Button-эталоне со своим маленьким rule-механизмом. Ядро `@capsuletech/rules` выделяем **после** эталона, когда формы правил докажут, что сходятся. Преждевременная абстракция = риск неправильного фундамента.

---

## 🪜 Инкрементальный путь

1. **Один rule** на Button (`accepts`/`isLeaf`) → редактор на него реагирует → доводим петлю до **эталона**.
2. Добавляем правила по одному → каждое = новые валидации/подсказки.
3. Тематические наборы (`data`/`api`/`test`) подключаются как новые rule-пакеты.
4. Учим контракты плагинам → дальше каждый новый контракт расширяет систему.
