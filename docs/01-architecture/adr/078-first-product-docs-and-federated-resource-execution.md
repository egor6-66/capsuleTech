# ADR 078 — Первый v2-продукт = docs; ярусы capability/composition; федеративное ресурсно-тарифное исполнение

## Контекст {#context}

ADR 077 задал v2 = дисциплинированная миграция + «первый продукт строится сразу
в v2» (там первым числился `moderator`). При проработке плана миграции (аудит
`docs/_meta/migration/`) вскрылись три вещи, сдвигающие стартовую точку:

1. **Доко-дисциплина — ровно та боль v1** (ADR 077 §Контекст п.2), которую надо
   закрыть раньше всего. Движок доков подвязан на Obsidian (wikilinks/callouts в
   `render-markdown.ts` + Prose-стили `.wikilink`/`.callout-*` — флаг аудита
   web-docs/web-ui U6); нет стандартов стилей/валидаторов; доку будут дополнять
   люди и агенты → рост корпуса = рост косяков. При этом куски уже есть: движок
   рендера, `backend/lang` с source-agnostic vault-import (просился standards-гейт
   dry-run), web-renderer. Это **консолидация, не greenfield**.
2. **`packages/web/workspace` (learn/studio) — не фреймворк.** Это композиция
   ИЗ фреймворка. Совпадает с аудит-флагом #3 «workspace-зона растворяется» и с
   `docs/_meta/commercial/repo-map.md`, где learn/studio уже числились отдельными
   ship-юнитами.
3. **Монетизация экосистемы = РЕСУРС, не фича** (уточнено user 2026-07-07). Юзер
   платит не за разблокировку возможности, а за то, на чьём и насколько мощном
   железе она исполняется. Это дозревание ADR 072 (федерация) в конкретную
   исполнительную модель.

## Решение {#decision}

### 1. Первый v2-продукт = **docs** (amends ADR 077 §3: moderator → docs)
Полноценный продаваемый продукт под зонтиком Omnifield; мы — юзер №0 (догфуд).
Причины: бьёт прямо в доко-боль v1; в основном консолидация готовых кусков;
de-Obsidian всё равно неизбежен (аудит-флаг). `moderator` сдвигается позже.

### 2. `framework/` = чистый фреймворк; продукты — снаружи (resolves аудит-флаг #3)
`framework/` = `runtime/ kit/ builders/ boost/ domain/ cli` в одном монорепо
(`@omnifield/*`, co-release — не дробить, version-hell). **learn / studio / docs /
moderator НЕ в нём** — это продукты (композиция), живут в своих репо. `zones.ts`
теряет зону workspace.

### 3. Три яруса
- **Framework** — монорепо `@omnifield/*`, чистый.
- **Capability-движки** (shared backend) — `docs`-движок, `lang`, `voice`, `llm`,
  `image`; группируются по capability-домену (как сейчас `backend/`),
  self-hostable (ADR 072). Потребляются МНОГИМИ продуктами через API.
  `gateway`(infra) / `identity`(auth) — не движки, живут отдельно.
- **Продукты** (composition) — docs/studio/learn/moderator, каждый свой репо;
  бэк+фронт продукта вместе (только СОБСТВЕННЫЕ части); потребляют framework (npm
  `@omnifield/*`) + движки (API).

### 4. Федеративное ресурсно-тарифное исполнение (кристаллизует ADR 072)
Деньги = ресурсный тариф, НЕ gate фич. Код возможности открыт везде; варьируется
железо/модель за ней. Для любой ресурсно-вариативной возможности юзер выбирает
исполнение: **наша общая бесплатная** (слабая модель) / **локально** на своём
железе (насколько тянет) / **Pro** (наши жирные модели на наших машинах) /
**peer** (чужие предложенные ресурсы). Двунаправленно — любой может предложить
свои ресурсы в общий пул.

Примитив:
- **Capability** — типизированный провайдер-агностик контракт (`generate`,
  `embed`, `tts`, …). Не знает, где исполнится.
- **Provider** — реализация на ресурсе: `{ capability, tier:
  free-shared | local | pro | peer, endpoint: inproc | remote-url, метадата
  качество/скорость }`.
- **Router** — выбор провайдера по entitlement + предпочтениям юзера
  («локально» / «дешевле» / «умнее»).
- **Entitlement-шов** — фильтр доступных пулов (local+free всегда; pro — по
  подписке; peer — по политике). **MVP = no-op: только local+free.**
- **Federation** — endpoint провайдера сетево-прозрачен (наш/пир) → «предложить
  ресурсы» = регистрация своей ноды провайдером в пуле.

Две породы возможностей:
- **Детерминированные** (parse / validate / render / локальный поиск / граф) —
  всегда локальные, БЕЗ роутера. Ядро продукта, едет внутрь, работает всегда
  бесплатно.
- **Ресурсно-вариативные** (llm / embed / tts / image) — только эти через
  provider/router. Тут живёт вся федерация и монетизация.

### 5. Что закладываем в базу СЕЙЧАС (дёшево; приклеить потом = переписывание)
Provider-агностик capability-контракты; сетево-прозрачный provider-интерфейс;
router + no-op entitlement-шов; capability-гранулярные (отдельно
монтируемые/гейтируемые) эндпоинты. **НЕ строим:** реестр пула, peer-маркетплейс,
биллинг, качество-тарифы, closed-модули. Реальных провайдеров на MVP — только
`local` + наш `free-shared`.

### 6. Репо-раскладка старта
- **`engines`** (бэк-монорепо capability-движков) — готовим фундамент со швом §5;
  первым заезжает **docs-backend**.
- **`docs`** (продукт) — обкатываем доку, эталоним структуру, первый догфуд
  commons. Фронт временный выкидной (тупой — только дёргает ручки бэка), реальный
  придёт из фреймворка позже; логика вся в бэке, чтобы замена фронта ничего не
  переписывала.

## Последствия {#consequences}

**Плюсы:** доко-боль v1 закрывается первой; первый продукт валидирует и
commons-дисциплину, и (по мере надобности) первую волну спайна фреймворка под
реальным юзкейсом; федерация/монетизация ложатся на готовый шов без
переписывания; docs MVP почти весь детерминирован → сетку строить ради старта
не нужно.

**Риски:**
- **Раздувание «платформы документации»** → митигация: MVP = детерминированное
  ядро + дремлющий provider-шов; ресурсно-вариативное (AI/озвучка) — отдельной
  волной по готовому шву.
- **Многорепо/многоязык с первого дня** (docs-фронт + питон-бэк + `engines`) —
  осознанно; граница capability/composition держит это управляемым.
- **docs-продукт тянет минимальный спайн фреймворка** (renderer/ui/core/style/…)
  в v2 — это не «строго после миграции», а первая волна под давлением реального
  потребителя (скорее плюс, чем риск: переносим не «по списку», а под нужду).

## Открытые вопросы {#open}
1. Кто заводит GitHub-репо под `omnifield` org (user руками / architect через
   `gh repo create`).
2. Рабочее имя docs-продукта (временное `omnifield-docs` → финал позже).
3. Гранулярность `engines`: один монорепо всех движков сейчас, split по
   capability-домену позже (не дробить преждевременно).
4. Нужен ли docs `lang`/`voice` на MVP — нет; ядро MVP детерминировано
   (ingest + валидация + рендер-дерево). AI/озвучка — перспектива.
5. Порядок первой волны спайна фреймворка под docs-фронт (какие `@omnifield/*`
   нужны, когда реальный фронт придёт из фреймворка).
