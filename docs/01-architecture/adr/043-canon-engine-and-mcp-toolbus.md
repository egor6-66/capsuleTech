# ADR 043 — Канон-движок как общий слой + MCP как tool-bus (CLI/MCP/агенты — тонкие обёртки)

## Контекст

1. **Скоро нужны эталонные аппы**, и юзеры будут собирать аппы не только через Claude, а через **локальные модели** (у нас уже есть инфраструктура подключения локал/ремоут моделей — `backend/scriber`).

2. **Агенты сейчас разбиты по слоям** (`.claude/agents/`: entity/view/shape/widget/controller/feature/page), и **канон каждого слоя живёт в промпте субагента** — то есть привязан к Claude, дублируется (общие HCA-правила в 7 местах), недетерминирован.

3. **MCP (Model Context Protocol)** — индустриальный стандарт «розетки» для инструментов: модель-агностичный, работает с любой моделью и любым клиентом. У нас в `backend/scriber/mcp` уже есть scaffold `MCPToolProvider` (**consumer-сторона** — наш агент подключается к внешним MCP-серверам). Producer-стороны (выставить *свои* инструменты) пока нет.

4. **CLI тащит в себе логику генерации** — это неправильно. CLI — лишь **один из способов** вызвать скрипты, а не место их реализации.

5. Пользователь хочет **крутилку «жёсткий шаблон ↔ фантазия»** для генерации — настраиваемый режим выхода.

## Решение

### 1. MCP — две стороны

- **Consume** (есть scaffold): `MCPToolProvider` втягивает чужие готовые инструменты (filesystem/git/fetch/memory из эталонного репо `modelcontextprotocol/servers` и реестра) в нашего агента. Не реализуем сами — берём готовое.
- **Produce** (новое): `capsule` MCP-**server** выставляет наши генераторы как tools. Тогда **любая** модель (локал/ремоут) собирает capsule-аппы, дёргая типизированные кнопки, а не выдумывая код наугад. Потребители сервера — и `scriber` (через `MCPToolProvider::from_stdio`), и Claude Desktop / Claude Code.

> Оговорка: инструменты модель-агностичны *механически*, но слабая локалка теми же кнопками примет решения хуже (порядок, ранний отказ, кривая склейка). Кнопки одинаковые — голова разной силы.

### 2. Канон-движок как отдельный общий слой (ядро)

Логику генерации **вынимаем из CLI** в отдельный пакет-движок. CLI, MCP-сервер и Claude-агенты становятся **тонкими ртами** поверх него — один движок, много входов:

```
        [ канон-движок: декларативный спек слоёв + рендер + крутилка ]
              ↑                ↑                  ↑
            CLI            MCP-сервер        Claude-агенты
        (терминал)       (любая модель)      (как сейчас)
```

Это тот же принцип, что и во всей capsule: логика в core, обёртки тонкие.

### 3. Канон слоя = декларативный спек (вариант «в»)

Где может жить канон — три варианта:
- **(а) текстом в промптах субагентов** (как сейчас) — гибко, но Claude-only, недетерминированно, дублируется;
- **(б) жёсткими генераторами-шаблонами** — предсказуемо, без модели, но никакой фантазии;
- **(в) декларативным описанием слоя** (данные: сигнатура wrapper'а, разрешённые импорты, пример, мета), которое умеют читать **и** шаблонизатор, **и** модель.

**Выбран (в)** — и его форсирует крутилка из п.5: чтобы ползунок скользил «жёстко ↔ фантазия», канон обязан лежать в нейтральной форме. На одном краю движок рендерит по шаблону (нужны данные), на другом — отдаёт те же данные модели «придумай». Пункты «где канон» + «крутилка» + «CLI без логики» сворачиваются в **одно решение**.

### 4. Первые tools MCP-сервера (замкнутый цикл, не «tool на слой»)

1. **`describe_layer(layer)`** — контракт слоя on-demand (пресет из п.3). Для мелких моделей с малым контекстом fetch-по-запросу лучше, чем 7 канонов в system-prompt.
2. **`inspect_registry()`** — что уже зарегистрировано (`Entities.*`/`Views.*`/`Widgets.*`, slot-имена, сигнатуры; читает `.capsule/registry`). Убивает галлюцинацию №1 — ссылку на несуществующий артефакт.
3. **`create_artifact(layer, spec)`** — детерминированно пишет файл по канону («руки» вместо freehand).
4. **`verify()`** — `capsule build` + compliance-линтер → структурированные ошибки (`file:line` + hint). Цикл: создал → проверил → починил.

Вместе: **describe → inspect → create → verify → (fix)** — весь агент-цикл сборки аппа как типизированные tool-call'ы.

**Порядок:** дешёвый первый шаг — `inspect_registry` + `verify` (чистое чтение существующего: CLI + registry + compliance уже есть, канон кодифицировать не надо, сразу дают любой модели «глаза + проверку»). `create_artifact` — вторым, после движка из п.2-3.

### 5. Где живёт сервер

**TS**, обёртка над `@capsuletech/cli` actions / канон-движком (генераторы TS-side, TS MCP SDK есть), транспорт stdio. Rust-сервер не оправдан: CLI всё равно тянет node.

## Альтернативы (отклонены / отложены)

- **Оставить канон в промптах субагентов** — Claude-only, не работает на локалках, дублируется. Отклонено как единственный механизм (Claude-агенты остаются *одним из* потребителей движка).
- **Реализовать filesystem/git/fetch самим** в `capsule-native-tools` — есть готовые эталонные MCP-серверы, берём их (consume-сторона). Свой код только под capsule-специфику.
- **1 агент + layer-пресеты vs 7 layer-агентов** (смежное обсуждение) — деталь реализации Claude-стороны; в этом ADR не фиксируется. Профит пресетов — maintainability + портативность, не «умнее генерит».
- **Эталонные аппы** — отложено: сперва поднять агентов и довести пакеты до идеала. Апп = «просто в нашем стиле с мок-логикой».

## Последствия

- **owner-cli + owner-shared (+ возможно новый пакет-движок):** вынос логики генерации из CLI в общий слой; CLI худеет до тонкого рта.
- **owner-scriber:** producer-сторона MCP (новый `capsule` MCP-server) + доведение `MCPToolProvider` (consume) из scaffold до рабочего.
- **Канон слоёв** нужно кодифицировать в декларативный спек (стоимость — основной вопрос; форсируется крутилкой).
- **Комплементарно с [[037-playground-capability-and-codegen-subgenerators|ADR 037]]:** 037 интроспектирует *созданные* артефакты (manifest), 043 их *генерирует*. Общий знаменатель — sub-generators / канон как данные.
- Архитектура (контракты движка, спек слоя, tool-набор) — главный (архитектор); реализация — owner'ы.
