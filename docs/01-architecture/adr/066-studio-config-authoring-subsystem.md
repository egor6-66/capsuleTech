# ADR 066 — Config-authoring подсистема студии (настройки, граф зависимостей, сериализация)

## Контекст {#context}

### Потребность {#need}

У приложения есть **роли**; у роли — **настройки**, влияющие на интерфейс и функционал.
Нужен визуальный редактор внутри студии, где:
- добавляем **настройку** (напр. флаг `showFilter`) с описанием «что и зачем»;
- настройку можно **привязывать** — к ролям или к другим настройкам/флагам;
- зависимости **валидируются на сборке**: добавили «варианты фильтра», но не включили
  `showFilter` → **ошибка** (нарушен `requires`);
- на выходе — конфиг **по шаблону**: XML / YAML / JSON.

Реализовать — внутри студии, как **отдельные модули**.

### Изоморфизм существующей студии {#isomorphism}

Config-авторинг структурно повторяет то, что студия уже делает для UI-композиции — это её
**второй домен**, а не фича поверх:

| UI-композиция (есть) | Конфиг-авторинг (ADR 066) |
|---|---|
| каталог манифестов компонентов | каталог **спек настроек** |
| `document` (дерево нод + пропсы) | **config-model** (выбранные настройки + значения + привязки) |
| Inspector (пропсы по zod + описание) | редактор настройки (значение + описание + привязки) |
| accept-policy / compliance | **граф constraints** (`requires` / `conflicts`) |
| канвас (проекция схемы) | **превью сериализации** (проекция модели → XML/YAML/JSON) |

Следствие: студия — **generic spec-driven authoring host**; переиспользуем Inspector и паттерн
«каталог → выбрал → правишь → валидируется → проецируется», не изобретаем механику заново.

## Решение (черновик) {#decision}

### D1 — Config = вторая авторинг-область студии {#D1}

Отдельный набор модулей студии (subpaths, зеркало composition-домена), НЕ смешивается с
UI-`document`. Своя модель, свой стор, своя проекция-превью.

### D2 — Разбивка на модули {#D2}

1. **`config/registry`** — каталог спек настроек: `{ id, type: flag|enum|number|string|group,
   label, description, deps }`. Точка «добавить настройку showFilter».
2. **`config/model`** — авторимый конфиг: включённые настройки + значения + привязки. Аналог
   `document.ts` (единый стор + selectedId + ops).
3. **`config/constraints`** — движок правил (`requires` / `conflicts` / `visibleWhen`),
   вычисляется на сборке → список ошибок.
4. **`config/inspector`** — правка настройки + привязок (переиспользует studio `Inspector`).
5. **`config/serialize`** — эмиттеры XML/YAML/JSON из одной модели, **pluggable per-format**
   (тонкие, как reporters профайлера — ADR 063).
6. **`config/roles`** — роль как измерение: матрица роль × настройка.

### D3 — Унифицирующая абстракция: conditions над symbol-table {#D3}

«Привязать к роли» и «привязать к другому флагу» — **один механизм**: **условие** над общей
таблицей символов (роли + настройки). `requires showFilter` = условие `showFilter == true`,
обязанное держаться. Role-binding = условие `role ∈ {...}`. Один слой **conditions/constraints**
покрывает зависимости, conflicts и role-привязки. Не плодим отдельные сущности «зависимость» и
«привязка».

### D4 — Валидация: build-time error-list (compliance-стиль) {#D4}

Per-setting значение — **zod** (переиспользуем движок валидации фреймворка). Cross-setting
зависимости — отдельный слой (`config/constraints`), вычисляется на сборке → список ошибок
(как compliance), НЕ блокирует правки в процессе. `requires`-нарушение = unmet-условие в списке.

### D5 — Сериализация: pluggable per-format {#D5}

Модель = plain data. Сериализаторы независимы per-format (JSON/YAML тривиальны; XML — своя
shape-конвенция). Добавление формата = новый эмиттер, не трогая модель.

### D6 — Фазировка (рекомендация) {#D6}

Развилка «что авторим»:
- **(A)** схему конфига — юзер создаёт сами настройки в UI (мета-редактор);
- **(B)** инстанс — настройки объявлены в коде, студия расставляет значения/привязки;
- **(C)** две фазы — MVP на (B) поверх code-declared реестра, затем (A) поверх того же контракта.

**Рекомендация — (C):** сначала вертикальный срез (реестр в коде → авторинг
значений/привязок/constraints/сериализации в UI → живой весь флоу), UI-создание самих спек —
второй фазой, когда контракт устоится. Не абстрагируем мета-уровень вперёд домена
(«прототип → продвижение», homework-before-abstraction).

## Переиспользование, не дубль {#reuse}

`Inspector` и паттерн document-стора переиспользуются напрямую. Соблазн вынести **generic
authoring-kernel** (каталог + model-стор + inspector + validate + serialize), где и
UI-композиция, и конфиг — инстансы, — **отложить**: извлекать общее ядро ПОСЛЕ того, как второй
домен реально заработает, а не абстрагировать вперёд (риск преждевременной абстракции).

## Открытые вопросы {#open}

1. **A/B/C** — финальный выбор фазировки (рекомендация — C).
2. **Цель вывода** — конфиг для **самого capsule-рантайма** (роли гейтят UI приложений) или
   **generic-артефакт** для внешнего потребителя? Влияет на shape XML/YAML и на связь с
   существующим `defineAppConfig`/`IAppConfig` (ADR 013).
3. Выразительность constraints MVP — предлагаю только `requires` + `conflicts` (кейс = `requires`),
   булевы выражения — позже.

## Последствия {#consequences}

- **+** Студия обретает второй домен на существующей механике; конфиги авторятся визуально с
  проверкой зависимостей и мульти-формат выводом.
- **+** Один слой conditions вместо трёх сущностей (deps/bindings/conflicts).
- **−** Новая подсистема = серия owner-брифов (owner-studio) + контракты; вес сопоставим с
  composition-доменом. Требует фиксации развилок до старта.
- **Связано:** ADR 013 (defineAppConfig), ADR 033 (регистрация), ADR 063 (тонкие
  provider/pluggable reporters — прецедент для D5).
