# ADR 041 — Модель композиции и дистрибуции (примитивы · инъекция · тогл/инжект · hub)

## Контекст {#context}

При пилении фич «по одной целиком» мы дублируем функционал (смена фокуса между фичами). Плюс нужна гибкая дистрибуция: один продукт под много заказчиков с разными фичами — **без пересборки фреймворка под каждого и без отдачи заказчику чужого кода**. Нужна единая модель: как строим и как раздаём.

## Решение {#decisions}

### 1. Композиция: большие фичи — ИЗ примитивов (декомпозиция-первой)

Супер-фича → декомпозиция в переиспользуемые примитивы (новый пакет / subpath-саб-пакет / расширение существующего) → строим/расширяем single-source → собираем фичу. Reuse/extend прежде create. Первый шаг архитектора по фиче — декомпозиция + проверка «что уже есть/шарится». Детали — [[compose-and-inject]].

### 2. Расширяемость: ИНЪЕКЦИЯ, не флаги

Пакет = абстрактная база + точки инъекции; специфичное **инжектится** (стратегия/генератор/конфиг), не включается флагом. Референс — codegen base+sub-generators (ADR 037). Механизм **гибкий** (конфигурируемый), не бинарный on/off. Примеры: `gen` injectable generators (ADR 040), auth-стратегии (ADR 039), будущий **logic-creator** рядом с ui-creator (общий creator-примитив — заложить при подходе).

### 3. Дистрибуция: тогл (роль) vs инжект (app/заказчик) — РАЗДЕЛЯЕМ

- **Фичатоглинг** — остаётся, но **только роль-уровень**: грубый доступ к фичам ВНУТРИ сборки заказчика (его же код, его роли).
- **Инжект** — всё app/заказчик-специфичное: **compose-time**, в артефакт едет ТОЛЬКО код заказчика (tree-shake выключенного). Никакого runtime-тогла со всем кодом → юзер не получает чужой код (security + вес).
- Источник правды композиции — **`capsule.app.ts`** (единая точка: пакеты, теги, перехваты роутинга, конфиги) + manifest (ADR 037). Внешняя настройка рулится отсюда, не отдельным config-хабом.

### 4. North-star: Docker-hub workflow

Целевая система (собирается из примитивов инкрементально):
```
Разраб → пушит мастер-сборки (со ВСЕМИ фичами, структурированные) → дев-воркспейс (Docker)
Саппорт → в UI (превью аппа + конфиги): тоглит роль-фичи / инжектит под заказчика
        → ПЕРЕСБОРКА под заказчика (compose-time инжект → только его код)
        → сохраняет пресет (не настраивать с нуля)
        → гоняет тесты/CI в Docker (видит консоль)
        → по зелёному: артефакт улетает на endpoint заказчика
```
Примитивы, из которых собирается hub: `shared-zod/gen` (данные), manifest/config-surface (ADR 037), playground/devtools-shell (UI превью+конфиг+тесты), Docker-runner (build/test/deploy), console-stream (ansi-to-html), test-runner (Vitest Node API + Playwright), build-time инжект-компоновка.

### 5. `web-bridge` — резерв имени

`@capsuletech/web-bridge` зарезервирован под **будущее** (WebAssembly, Tauri-invokers, нативный мир). НЕ config-хаб, сейчас не реализуем.

## Последствия {#consequences}

- Флоу [[compose-and-inject]] вписан в `docs/_meta/architect-routing.md` как правило decision-making.
- Каждый примитив hub'а — **свой ADR при подходе** (gen=040; runner/deploy/hub-UI — позже).
- Дистрибуция (инжект-компоновка, роль-тогл) — отдельный ADR при реализации; ремоуты (ADR 015) — кандидат для runtime-инжекта, но базовый путь — compose-time.
- `logic-creator` — будущий пакет; общий creator-core с ui-creator (декомпозиция при старте).

## Альтернативы (отклонены) {#alternatives}

- **Пилить фичи монолитом** — дубль функционала. → композиция из примитивов.
- **Флаги/фичатоглинг для всего** — тащит чужой код, не масштабируется. → инжект (тогл только роль-уровень).
- **Отдельный config-хаб-пакет** — раздув; настройка и так в `capsule.app.ts`. → стандартные механизмы.
