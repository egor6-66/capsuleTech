# ADR 061 — UiProxy: динамический ctx маршрутизации событий

## Контекст {#context}

### Pain 1 — `Ui` биндится к ctx точки СОЗДАНИЯ, не позиции {#pain1}

`View`/`Widget`-обёртки строят прокси UI-kit'а через `UiProxy(BaseUi, useCtx(), wrapperProps)` (`view.tsx:23`, `widget.tsx:27`). `useCtx()` возвращает ближайший логик-враппер-**предок самого View/Widget**, и этот ctx **захватывается** в `wrapComponent` (`ui-proxy.tsx`). Каждый `Ui.Button` из этого прокси диспатчит события в `ctx.controller`, зафиксированный на момент рендера обёртки.

Следствие: meta-узел маршрутит `onClick` в логик-враппер **над виджетом**, ИГНОРируя любую `Feature`/`Controller`, вложенную ВНУТРЬ тела этого виджета. Кнопка из инлайнового `Ui` виджета не доходит до фичи, которую виджет сам же отрисовал.

### Pain 2 — Это ломает `next()`-бабблинг {#pain2}

Вся остальная модель построена на «событие всплывает к ближайшему обработчику» ([[008-hybrid-fsm-api|ADR 008]] — `next()` делегирует родителю). Но захват ctx нарушает это для инлайнового `Ui`:

```tsx
Widget((Ui) => (
  <Controllers.A>
    <Controllers.B>
      <Ui.Button meta={{ tags: ['x'] }} />   {/* перепрыгивает A и B → предок виджета */}
    </Controllers.B>
  </Controllers.A>
));
```

Автор обернул кнопку в два контроллера — **ни один не ловит**. Событие уходит мимо всей вложенной цепочки. Это противоречит бабблингу и не имеет обоснования в коде.

### Pain 3 — Тихий провал, неинтуитивно {#pain3}

Провал **молчаливый**: ни ошибки, ни предупреждения — просто ничего не происходит. Причину невозможно понять без чтения движка (incident 2026-06-26: и architect, и user не понимали поведение, пока не залезли в `ui-proxy.tsx`). Захват не был осознанным решением — ни одного комментария/теста, фиксирующего «биндимся к виджету, а не к вложенной фиче»; сигнатура `buildEventBindings(ctx, …)` нейтральна. Это побочка оптимизации (прокси строится один раз), а не дизайн.

## Решение {#decision}

### D1 — Динамический ctx по позиции рендера {#D1}

`wrapComponent` → `ComponentWrapper` (рендерится на DOM-позиции элемента) резолвит ctx **в момент своего рендера**:

```ts
const liveCtx = useContext(Context) ?? ctx;
```

`useContext(Context)` возвращает ближайший логик-враппер-предок **самого элемента** (логик-врапперы провайдят свой ctx через `<Context.Provider value={ctx}>` в `logic-wrapper.tsx`). Захваченный `ctx` остаётся fallback'ом, когда внутреннего враппера нет.

Канон: **компонент шлёт события в логик-враппер, который его ОБОРАЧИВАЕТ (ближайший предок по дереву рендера), и никогда — в тот, что создал его `Ui`.** В обычном случае (нет вложенного враппера) `liveCtx === ctx` — поведение тождественно прежнему.

### D2 — store-владение скоупится к ближайшему врапперу {#D2}

`liveCtx` используется не только для event-binding, но и для регистрации (`registerComponent`/`unregisterComponent`), `updateComponent`, чтений `store.styles`/`store.props`. То есть meta-компонент принадлежит store'у **ближайшего оборачивающего** логик-враппера. Это семантически согласовано с D1 (один владелец — и события, и состояние) и тоже идентично прежнему в общем случае.

### D3 — `bindEvents` (composite-строки) остаётся на captured ctx {#D3}

Path `bindEvents` (внутренние строки composite-примитивов — `DataTable.Row`, `List.Item`) **намеренно** не переводится на динамику. Composite-строки рендерятся внутри того же контроллера, что владеет таблицей/списком — они не пересекают границу логик-враппера. Captured ctx здесь корректен и проще. Если появится реальный кейс composite под вложенным контроллером — отдельный ADR.

## Последствия {#consequences}

### Положительные
- Интуитивно: фича/контроллер ловит события своих потомков независимо от того, где создан их `Ui`. Фича может оборачивать виджет ИЛИ жить внутри него — обе формы работают.
- Восстановлен `next()`-бабблинг для инлайнового `Ui` (вложенные контроллеры больше не перепрыгиваются).
- Снят молчаливый footgun (Pain 3).
- Backward-compat: общий случай (нет вложенного враппера) → `liveCtx === ctx`. Все 514 web-core тестов green (+ новый nested-ctx тест, ранее поведение не покрывалось вовсе).

### Отрицательные / открытые
- `useContext(Context)` на каждый рендер meta-компонента — стоимость пренебрежимо мала (один контекст-lookup).
- Нельзя «прокинуть» событие инлайнового `Ui` мимо вложенной фичи к предку виджета — теперь вложенная фича его перехватит. Это и есть желаемое поведение; нужен предок — не вкладывай фичу.
- `bindEvents` и `wrapComponent` разъезжаются по модели (D3) — осознанный trade-off, обоснован отсутствием cross-controller composite-кейсов.

## Валидация {#validation}

- **Browser (реальный кейс):** `Features.Canvas` внутри тела виджета playground — клик кнопки внутри неё теперь диспатчит в `Features.Canvas.onClick` (раньше — молча мимо).
- **Unit:** новый тест — meta-кнопка под вложенным вторым ctx-Provider'ом → событие идёт во внутренний контроллер. Суита 512 → 514, всё green.
- Дифф — 5 точек в `wrapComponent`, без касания публичного API.
