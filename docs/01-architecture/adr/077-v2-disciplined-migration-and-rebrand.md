# ADR 077 — v2: дисциплинированная миграция в новый репо/бренд (не rewrite)

**Status:** proposed (обсуждается, 2026-07-06)
**Deciders:** user (product owner) + architect
**Companion:** нейминг/бренд (`docs/_meta/commercial/`), ADR по package-канону (см. [[feedback_package_equals_app_layers]] — «пакет = апп минус Page/Feature», тестбед moderator).

---

## Контекст

Проект вырос из «веб-фреймворка с инкапсуляцией слоёв» в полноценную экосистему (HCA-фронт + kit + CLI + backend capability-сервисы voice/image/llm/lang/learn + studio + apps learn/community/moderator + Unity-амбиции). Фундамент **хороший** — фичи ложатся «как влитые». Но накопились проблемы, и они **не в коде, а в процессе/организации**:

1. **Нет роадмапа/стадий** → много «подвешенных» состояний; никто (включая овнеров) не всегда понимает, на каком этапе что. Кейс-пример: уроки залили → решили менять → валидатор пишем на ходу → БД/фронт не готовы → контент льётся мимо валидаторов. Причина — нет Definition of Ready по стадиям.
2. **Доки не структурированы** — дубли, противоречия, актуальность определяется по датам; важное уезжает в угол, всплывает только после граблей.
3. **Нет стандарта интеграций** — внешние скоупы (учитель, Unity-архитектор) общаются обменом брифов в папке; каждая интеграция болезненна, овнеры внешних скоупов теряются «с кем имеют дело».
4. **Рефактор фундаментальных вещей дорог** — пакеты расплодились без эталона; правки требуют огромных волн.
5. **Имя `capsule`** придумано для web-only-эры; для экосистемы не в тему; домен занят; в поиске ассоциируется с аптеками/таблетками.

## Решение

**Поднять v2 в новом репозитории под новым брендом/аккаунтом и переносить проверенное итеративной МИГРАЦИЕЙ, доводя каждый кусок до эталона на въезде. Это НЕ rewrite-с-нуля.**

Ключевые принципы:

### 1. Миграция, не переизобретение
Старый `capsule`-репо → статус **«черновик / оракул»**: новые фичи туда не пишем, только вычитываем при переносе. Каждый кусок едет по циклу **копия → верификация против оракула → эталон → доки → трейсы**. Логику по памяти НЕ переписываем (это классическая ловушка rewrite'ов — ×2–3 времени, возврат пофикшенных багов, потеря momentum, и новый репо наследует ту же недисциплину).

### 2. Дисциплина ставится ПЕРВОЙ (иначе v2 зарастёт как v1)
Настоящий фикс — процесс, не папки. В v2 сразу:
- **Роадмап + стадии** — один живой файл статуса по фичам; **Definition of Ready**: фичу не начинают вниз по потоку, пока не готов апстрим (флоу/БД/валидатор/фронт).
- **Эталон-гейт на пакет** — «готов» = код + тесты + трейсы + доки + эталонная раскладка (канон [[feedback_package_equals_app_layers]]).
- **Гигиена доков** — один источник правды на тему, явная пометка `superseded`, без дата-археологии.
- **Протокол интеграций** — стандарт для внешних скоупов: онбординг «с кем имеешь дело» + версионируемый контракт интерфейса + канал общения (не «брифы наугад»).
- **CI/тесты/овнерство** — переносим с выученными граблями, не воспроизводим их.

### 3. Первый продукт строится сразу в v2 (объединяющий ход)
`moderator` (первый релиз для модераторов/тестеров) НЕ строим в старом capsule — строим **в v2 на валидированном package-каноне**. Так в один заход совпадают: ребилд-подложки + первый релиз + валидация канона + ребренд + новый аккаунт/домен/CI. v2 — не детур в ущерб релизу, а место, где первый продукт делается правильно.

### 4. Порядок переноса
Сначала «влитой» фундамент: **core / builders / kit / cli** (проверенные, стабильные) → затем домены **по одному**, эталонясь на въезде. Backend capability-сервисы (lang/voice/image/llm/learn/auth/community) — переносятся как готовые контракты (ADR 067 топология сохраняется).

### 5. Бренд/аккаунт
Новый репо живёт под новым GitHub-org/Claude-аккаунтом с новым именем (нейминг — companion-решение, см. `docs/_meta/commercial/`). Переименование in-place НЕ делаем — v2-граница и есть момент ребренда. Локальный контекст (`~/.claude` память/хуки/агенты) файловый и переносимый — переезд не теряет накопленное.

## Последствия

**Плюсы:** чистая подложка под экосистему; дисциплина с нуля; ребренд без боли in-place; первый продукт = первый гражданин v2; старый репо как безопасный оракул.

**Минусы/риски:**
- **Ловушка rewrite'а** — снимается жёстким правилом «мигрируем, не переизобретаем» + оракул.
- **Двойной фронт** — период, когда живут оба репо; митигация: ровный темп миграции, заморозка новых фич в capsule (кроме доведения переносимого), не «два фронта навечно».
- **Дисциплина следует за людьми** — если не поставить процесс-правила, v2 зарастёт; поэтому §2 — не опционален.
- Время: инвестиция сейчас ради «потом как по маслу» (осознанное решение user).

## Открытые вопросы
1. **Имя/бренд** — companion-сессия нейминга (позиционирование → кандидаты → доступность домена/npm/org → шорт-лист). Блокирует «куда переносить».
2. **Порядок**: добить moderator-POC в capsule (дешёвая проверка механики канона) → v2-скелет параллельно → moderator-продукт уже в v2. (Предложение architect'а.)
3. **Feature-freeze capsule** — с какого момента; доживаем ли текущие волны.
4. **Что именно = «оракул-заморозка»** vs что доводим в capsule перед переносом (подвешенные: дриллы, moderator-флоу, content-валидаторы, auth-хвосты, unity ф.0).
