# ADR 063 — Тонкие провайдеры, capability-субпаты (Provider не барел)

## Контекст {#context}

### Pain 1 — Provider стал барелом {#pain1}

`@capsuletech/web-profiler` `ProfilerProvider` (`providers/profiler.tsx`) статически импортит **три разных концерна**: trace-канал (мелочь), **13 perf-коллекторов** (web-vitals/fps/longTasks/observers — тяжёлые, непрерывный overhead) и **`ProfilerDashboard`** (kobalte Tabs + панели — тяжёлый UI). Чтобы получить **одну** возможность (trace-логи), потребитель монтирует Provider — и тащит в бандл остальные две. Это **барел под видом провайдера**.

### Pain 2 — потеряна независимость возможностей {#pain2}

Нельзя взять только `trace`, не заплатив perf+UI. Прямой кейс «в dev нужен максимум, в prod — только логи» нерешаем: trace заперт за флагом `vitals`, который гейтит **весь** тяжёлый профайлер → в prod с `vitals:off` trace включить нельзя, его кода нет в бандле. Возможность, которая самодостаточна, не раздаётся самодостаточно.

### Pain 3 — recurring, и «временное становится постоянным» {#pain3}

Это **не первый раз** (барел/связка). Без зафиксированного правила каждый пакет рискует тем же: Provider обрастает всем подряд, и «то, что начиналось временным, остаётся навсегда». Quick-fix (поднять весь тяжёлый Provider ради дебаг-логов) — долг, несовместимый с миссией (канон §0).

## Решение {#decision}

### D1 — Capability = независимый субпат {#D1}

Каждая **самодостаточная** возможность пакета раздаётся через **собственный export-субпат**, импортящий только то, что нужно ЕЙ — без втягивания соседних capability. Внутренние хелперы остаются приватными. Чтобы заюзать X, потребитель не тащит Y.

### D2 — ОДИН тонкий Provider = оркестратор-хаб {#D2}

Provider — **один** (рут, всегда смонтирован). Несёт **только свою логику**: шину/контекст + приём-роутинг сигналов от сабмодулей. **Не импортит ни одного сабмодуля** (ни коллекторов, ни UI, ни reporters). Ловит сигналы продюсеров (trace, collectors), раздаёт читателям (Dashboard, кастомный логер, beacon) — какой интерфейс мониторинга подключён, решает потребитель.

**Сабмодуль ≠ prop:**
- **prop = конфиг** — настройка того, что УЖЕ есть в провайдере (`capacity`, `historySize`, toggle).
- **сабмодуль = children** — независимый плагин, цепляется через контекст и может жить **где угодно в дереве, в т.ч. в глубине** (не обязан стоять рядом с провайдером). Поэтому коллекторы/Dashboard/логеры — дочерние компоненты, само-регистрирующиеся через контекст, а НЕ props.

То, что capability работает только внутри Provider'а (общий контекст) — норм. **Барел-Provider — нарушение.**

### D3 — Комбо = opt-in композиция {#D3}

Пакет МОЖЕТ отдать удобную композицию («суперфункцию»), монтирующую несколько capability вместе. Но это **opt-in**: кто хочет одну возможность — берёт её индивидуально (её субпат / тонкий провайдер), **никогда** через комбо. Комбо не тянет лишнее в тех, кто его не выбрал.

### D4 — Потребитель композит под свой билд {#D4}

dev = максимум; prod = только нужное (напр. логи) → ненужные модули не грузятся и **не попадают в prod-бандл** (явный выбор субпатов / tree-shake). Билд платит за то, чем пользуется.

### D5 — Профайлер — первый эталон, потом аудит {#D5}

Применяем к `web-profiler` **первым** (de-barrel `ProfilerProvider`): из провайдера убрать импорты 13 коллекторов + presets + `ProfilerDashboard` — остаётся шина + trace-sink + контекст. `trace` / каждый collector / Dashboard / reporters — независимые субпаты-**сабмодули**, подключаются как **children** (через контекст), могут стоять в глубине дерева. `BaseProviders` композит: **один тонкий `ProfilerProvider` всегда** (ловит trace везде, тогл off, ноль оверхеда), collectors+Dashboard — opt-in дочерние сабмодули за `vitals`. Обкатываем флоу на профайлере, затем — **аудит остальных пакетов** под этот канон (по мере времени).

> Context-зависимость capability от своего Provider'а допустима. Возможный future-обход (context-free шеринг, напр. Zustand) — паркован, обсуждается на state-слое.

## Последствия {#consequences}

### Положительные
- Возможности независимы: взять X не тащит Y; observability доступна везде дёшево, prod не платит за коллекторы/UI.
- dev/prod-дифференциация билда решаема выбором субпатов.
- Снят recurring барел-риск: канон, против которого аудитятся пакеты.
- Эталон (профайлер) как образец для остальных.

### Отрицательные / открытые
- Больше субпатов = больше build-entries + `exports` на пакет (дисциплина multi-entry).
- Комбо-провайдер надо проектировать так, чтобы он не реимпортил всё для тех, кто его не выбрал.
- Аудит существующих пакетов под канон — отдельная продолжающаяся работа (после эталона).

## Валидация {#validation}

Эталон-критерий профайлера:
- Импорт `@capsuletech/web-profiler/trace` тянет **только** trace (без коллекторов и Dashboard) — проверяемо по чанку.
- `BaseProviders` монтит тонкий trace всегда + perf/Dashboard opt-in.
- Бандл потребителя «только trace» **не содержит** коллекторов и Dashboard.
