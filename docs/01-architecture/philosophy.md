# 🌑 Философия — UI is a Shadow

> [!quote]
> Интерфейс — это немая проекция логики. UI ничего не знает, ничего не решает, ничего не хранит. Это тень, отбрасываемая Controller'ом.

## Зачем это нужно

Классический подход (React/Vue/Solid в его «обычном» виде) **смешивает** три вещи в одном компоненте:

1. **Что показать** (JSX, стили).
2. **Как реагировать на ввод** (onClick-хэндлеры, локальный стейт).
3. **Что делать с миром** (fetch, навигация, валидация).

Это удобно для прототипа, но даёт цепную реакцию: чтобы изменить кнопку — надо понимать форму; чтобы вынести форму — надо переписать страницу.

HCA разделяет эти три уровня **физически**, в трёх разных файлах разных слоёв:

| Что | Где | Знает только про… |
|---|---|---|
| Что показать | `Entity` | Solid + типы |
| Как реагировать | `Controller` | FSM + meta-теги детей |
| Что делать с миром | `Feature` | API + services |

И связывает их **не импортами**, а композицией в `Widget`.

## Три ключевых принципа

### 1. Tell, don't ask

Entity не запрашивает состояние («а я disabled?» «а какой класс?»). Controller сам **сообщает** Entity её роль: `class`/`onClick`/`name` подменяются через [[ui-proxy|UiProxy]], а адресные команды (например «эта кнопка `disabled`») — явным `store.patch([tags], { disabled: true })`. Entity рендерит то, что ей дали. Заметь: `disabled` это **explicit tell**, а не глобальная магия — Controller адресно говорит конкретному компоненту, что он заблокирован.

### 2. Локальность принятия решений

Решения принимаются на ближайшем слое, который **обязан их знать**. Кнопка submit не должна решать, что делать. Это решает Form-controller. Form-controller не знает, как авторизовать. Это знает Auth-feature.

Если решение «пробивает» уровень — это сигнал [[controller-proxy|next()]].

### 3. Вырываемость (extractability)

> Любой компонент любого слоя должен быть **вырываем**: переноситься в другой проект, заменяться альтернативной реализацией, без шлейфа из соседних файлов.

Поэтому запрещены:
- горизонтальные импорты (Entity ↔ Entity, Controller ↔ Controller),
- знание о конкретике (Entity не знает имени API; Controller не знает url; Feature не знает класс кнопки).

## Что HCA **не** обещает

- Это не «быстрее» React'а. Профит — не в перформансе.
- Это не «меньше кода». На простой кнопке HCA проигрывает классике в 3–5 раз по строкам. Профит появляется на масштабе.
- Это не «универсально». Для одностраничных лендингов HCA избыточен. Целевая аудитория — длинноживущие приложения с большим количеством состояний и переиспользуемых форм.
