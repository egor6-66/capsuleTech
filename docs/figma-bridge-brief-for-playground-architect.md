# Бриф: Figma-прототип → JSON → рендер (для архитектора playground)

> ⏳ **Временный handoff-документ.** Прочитать, обсудить, удалить. Не часть постоянной доки.

## 🎯 Цель одной фразой

Дизайнер собирает прототип в **Figma** → из него получаем **JSON** → мы по этому JSON **рендерим живой UI**, и он выглядит **как в Figma**. Дизайн ведёт, код отрисовывает 1:1.

«Как в Figma» получается **не копированием пикселей**, а потому что и в Figma, и в рантайме — **один и тот же UI-кит и одни и те же токены**. Figma и код говорят на одном языке.

## 🧩 Почему мы для этого готовы (фундамент уже есть)

- **`@capsuletech/web-renderer`** — рантайм, который рендерит UI **по JSON-схеме**: принимает `ISchema` (дерево `IEditorNode` + опциональные `IInteraction`) и `Registry` (компоненты по dot-path, напр. `ui.Button`) → эмиттит Solid JSX. То есть «рендер по JSON» — это **уже работающий механизм**, не надо изобретать.
- **`@capsuletech/web-ui-creator/manifests`** — реестр спецификаций компонентов (props, можно ли вкладывать дочерние и т.д.). Это контракт, который описывает наш кит.
- **Registry** — компоненты по dot-path (`ui.Button`, `ui.Card`…), которыми оперирует renderer.
- **Токены уже канонизированы и подготовлены для Figma** (ADR 042 + бандл `docs/figma-handoff/`): набор = канон shadcn/Tailwind, синкается в Figma через Tokens Studio. Набор **заморожен** — работаем тем, что есть.

Недостающее звено — только **экспортёр Figma-фрейма в наш `ISchema`**.

## 🔄 Полный поток

```
1. PUSH: наши компоненты → Figma как библиотека (ui.Button, ui.Card, ui.Input …)
2. Дизайнер собирает прототип ИЗ ЭТИХ компонентов (не freeform-рисование)
3. PULL: Figma-плагин обходит выделенный фрейм
        → узнаёт инстансы наших компонентов
        → читает variant-props + Auto Layout
        → эмиттит наш ISchema JSON
4. web-renderer рендерит ISchema → живой UI = как в Figma
```

## ⚠️ Критическое условие

1:1 получается **только если дизайнер собирает из НАШИХ компонентов** (Figma-библиотека = наш registry). Если рисует произвольные прямоугольники/текст — плагину нечего маппить в `ui.Button`, выйдет мусор с абсолютным позиционированием.

→ Отсюда строгий порядок: **сначала PUSH** (наши компоненты как Figma-библиотека, чтобы было из чего собирать), **потом PULL** (плагин читает инстансы → JSON).

## 🛠️ Что надо построить

1. **PUSH — компоненты в Figma.** Либо готовое (story.to.design из нашего Storybook), либо свой плагин-генератор на `web-ui-creator/manifests`. Цель — чтобы у дизайнера была палитра наших реальных компонентов с вариантами.
2. **PULL — Figma-плагин-экспортёр** (TypeScript на Figma Plugin API). Основная новая разработка: обход дерева нод, распознавание инстансов наших компонентов, чтение props/layout, сериализация в `ISchema`.
3. **Слой маппинга** (главные решения):

| Figma | → | Наше |
|---|---|---|
| Auto Layout | → | `ui.Flex` / `ui.Grid` (ложится чисто) |
| Абсолютное/freeform позиционирование | → | плохо маппится — приучать к Auto Layout |
| Variant-props компонента | → | props нашего компонента (нужна конвенция именования, типа Code Connect) |
| Prototype-связи (клик → фрейм) | → | опционально `IInteraction` (простая навигация) |
| Цвета/токены | → | уже общие (канон-токены, ADR 042) |

## 🚧 Честные границы (важно проговорить с дизайнером)

- **1:1** — для компонентных, Auto-Layout макетов. ✅
- **Лоссово** — freeform-фигуры, кастомная графика, сложная анимация. ⚠️
- **Логика/состояние — НЕ переносятся.** Прототип = верстка + статика + простые переходы. App-логика (валидация, API, FSM) живёт в коде (Controller/Feature). JSON несёт структуру и презентацию, не поведение.
- Плагин-экспортёр — это **недели разработки**, не «на выходных».

## 📌 Где это в плане

- Это **ADR 043** (Figma↔code мост), пункт «Figma→code». ADR пока не написан — после согласования цели я (главный архитектор) оформлю контракт: `Figma-frame → ISchema` (маппинг компонентов/layout/props, scope/не-scope) + фазы (push-библиотека → pull-плагин → маппинг).
- Связанное: ADR 042 (канон-токены, **сделано**), бандл `docs/figma-handoff/` (токены для дизайнера, **готов**).
