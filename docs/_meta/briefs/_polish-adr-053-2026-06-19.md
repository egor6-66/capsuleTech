# Контекст

Прочитал ADR-053 полностью. Архитектурно — **подписываюсь под decisions 1-8**. Two-channel contract (props vs config), structured context в bootstrap, reserved-props classes, 6 validation checks в demo, Phase 1/1a split — всё canonical.

Ниже — **7 уточнений implementation-guidance / sequencing**, не disagree с decisions. Преимущественно additive правки в decisions / risks / roadmap. Если author согласен — стоит включить в next revision ADR'а перед тем как architect обновит бриф под ADR-053 и делегирует owner-web-remote.

# Уточнения

## a) `startApp(root, opts?)` — где живёт?

**Decision 1** говорит: «`startApp(root, opts?)` — общий helper, который инстанцирует Solid root + BaseProviders + RouterProvider + читает IAppConfig». Где именно живёт этот helper — не сказано.

**Два варианта:**

1. **Per-app частный код** (`apps/<name>/src/startApp.ts`). Symmetry чистая, app сам владеет своим lifecycle'ом.
   - Минус: **копипаста по умолчанию** между apps. При втором remote-capable app'е dev уйдёт в copy-paste.
2. **Canonical helper `createCapsuleApp({ root, config, eventSink? })`** в `@capsuletech/web-core/bootstrap` (новый subpath).
   - Плюс: zero duplication; `main.tsx` + `standalone.ts` каждый = 3-5 строк.
   - Минус: zone expansion в `owner-web-core` — но Phase 1a, не блокирует Phase 1.

**Рекомендация:** добавить **второй readiness checkpoint** в risks#7 рядом с `useAppConfig({ override })`:

> 7a. **`createCapsuleApp` canonical helper.** Симметрия standalone/embedded требует общего bootstrap-кода. В Phase 1 demo-apps пишут `startApp` руками; Phase 1a canonization — `owner-web-core` добавляет helper в `@capsuletech/web-core/bootstrap` subpath. Без этого второй remote-app получит копипасту.

## b) Merge order — host-side, прописать явно

Decision 3 фиксирует **порядок** merge'а (provider → module → instance), но не место. По коду в decision 4 видно: `{ ...modules[name]?.config, ...props.config }` — это host-side `createEffect` внутри `RemoteComponent`. Provider.config тоже host-side читается.

**Это implicit, но снимет потенциальный вопрос «должен ли module-side применять `<Remote config>` поверх своего default'а».**

**Рекомендация:** добавить в Decision 3 одну строку после merge order table:

> Merge применяется **host-side** в `RemoteComponent` (`createEffect`); module-side получает finalized snapshot через `__capsule_remote_config__` envelope и хранит его в `configStore`. Никакой merge-логики внутри iframe нет.

## c) Shell — boot.js, не inline srcdoc (sizing recommendation)

Decision 4+5+6 наращивают shell-логику:
- два Solid `createStore` + `reconcile`,
- два envelope handler'а (`__capsule_remote_props__`, `__capsule_remote_config__`),
- proxy fabric для props/config,
- on-message dispatcher с фильтром `(from, fromInstance, eventName)`,
- request/response pending Map,
- ready-handshake state.

ADR consequences-negative фиксирует: «при превышении удобного размера — выносить в boot.js». Это **выбор позже**.

**Рекомендация:** sizing-wise Phase 1 shell **сразу превышает** удобный inline-size. Сделать boot.js **дефолтом Phase 1**:

- **Debuggable URL** в DevTools (sources tab).
- **Type-checked** на build'е web-remote'а (`shell.ts → shell.js` через tsc/rolldown).
- **HMR** при разработке самого shell'а.
- **Не съедает символьный budget srcdoc** (escape JSON-stringify x N).
- **Cache'ит** между instance'ами (один HTTP запрос на shell, N iframe'ов).

srcdoc остаётся для **bootstrap-параметров** (NAME / INSTANCE_ID / SESSION_ID / ENTRY) — короткий inline-template который импортит `boot.js` как `<script src="/__capsule_remote_boot__.js">`. Это **не decision-change**, но **рекомендация** в Phase 1 acceptance / OWNERSHIP.md.

## d) Renderer-as-remote depends-on DnD-iframe ADR (explicit sequencing)

Risks#3 фиксирует:
> DnD через iframe boundary — pointer events не пересекают frame. Studio palette drag → renderer canvas drop **не работает** в Phase 1 архитектуре. Это **следующий архитектурный ADR**.

Roadmap Phase 1a фиксирует:
> Renderer-as-remote — `@capsuletech/web-renderer` экспозит `bootstrap()`; studio creator-mode переезжает на `<Remote name="renderer">`.

**Проблема:** studio creator-mode **без DnD = неработающий creator-mode** (drag-from-palette → drop-into-canvas — fundamental gesture). Phase 1a renderer landing **должен** ждать DnD-ADR (или приёмлемый workaround вроде host-overlay drop-zone).

**Рекомендация:** в roadmap Phase 1a добавить explicit sequencing:

> Renderer-as-remote + studio creator-mode переезд требуют **либо** DnD-через-iframe ADR (отдельный архитектурный документ, мотивируется renderer-как-remote сценарием), **либо** host-overlay drop-zone workaround (host рисует drop-zone поверх iframe, координаты mapped в iframe-relative через postMessage). Architect эскалируется при landing'е Phase 1, не делегируется в owner-web-renderer.

## e) Bootstrap-сигнатура — breaking-change vs текущий бриф

`web-remote-phase1-renderer-mvp.md` Phase 1.a / Phase 1.c шаблон srcdoc показывают **старую** сигнатуру:

```ts
export const bootstrap: IRemoteBootstrap = (root, props, channel) => { ... };
// внутри srcdoc-shell:
bootstrap(document.getElementById('capsule-remote-root'), msg.payload, channel);
```

ADR-053 decision 2 даёт **новую** structured-context сигнатуру:

```ts
export interface IRemoteBootstrap<Props, Config> {
  (root: HTMLElement, ctx: { props: Props; config: Config; channel: IRemoteChannel }): IRemoteDispose;
}
```

**Action для architect (не для ADR-author):** обновить бриф **полностью под ADR-053** перед делегированием в owner-web-remote — phase 1.a (types), 1.c (srcdoc/boot.js shell), 1.h (demo) переписываются под new signature + two-channel contract. Это не ADR-правка, а architect's todo.

## f) Multi-instance demo check 5 требует explicit `instanceId`

Demo check 5:
> Два instance'а `<Remote name="hello" instanceId="a" config={{ apiUrl: 'A' }}>` + `<Remote name="hello" instanceId="b" config={{ apiUrl: 'B' }}>` — каждый видит свой `apiUrl`.

Это **implicit** требует явный `instanceId` — без него `createUniqueId()` генерит разные id'ы, но host не сможет адресовать конкретный instance через `remote('hello', '?')` для отправки команды.

**Рекомендация:** в decision 7 (demo) **зафиксировать**:

> Multi-instance check требует **explicit `instanceId`** на JSX-узле. Auto-id (`createUniqueId()`) допустим только для случаев когда host не адресует instance явно через `remote(name, id)`.

## g) `config` shape в Phase 1 — `Record<string, unknown>` (явно)

ADR оставляет typed `config` shape на Phase 4 (manifest codegen). В Phase 1 — runtime-only, как и props.

**Рекомендация:** в Phase 1 Acceptance gates добавить одну строку:

> `IRemoteModuleConfig.config?: Record<string, unknown>` и `IRemoteComponentProps.config?: Record<string, unknown>` — untyped в Phase 1; runtime validation добавляется в Phase 4 через manifest codegen (zod-to-json-schema + TS generated types).

## h) `startApp` event-routing — как именно `eventSink` подменяет `useEmit`

Decision 1 говорит «`useEmit(...)` маршрутизируется либо в локальную app-event-bus (standalone), либо через channel наверх (embedded) — решение принимается `eventSink`-обёрткой в `startApp`». **Механизм подмены не описан.**

`useEmit` сейчас — canonical pattern domain-пакетов ([[041-composition-distribution-model|ADR 041]]). Чтобы HCA-слои app'а не знали режима, `startApp` должен либо подменять implementation `useEmit` через Provider (по аналогии с `RouterProvider`), либо инжектировать routing-фабрику в `BaseProviders`. Без этого Решение 5 (`on*` props автосаб) работает только с искусственно `channel.send(...)`-ующими модулями; реальные app'ы остаются на manual wire'е.

Если в Phase 1 demo-`hello` шлёт через `channel.send('clicked', ts)` напрямую (минимальный module с одной кнопкой, не полноценный HCA-app) — это **OK для validation**, но надо явно сказать что **HCA `useEmit` → channel routing** = Phase 1a backfill, координируется через `owner-web-core`.

**Рекомендация:** добавить в Решение 1 после параграфа про `useAppConfig`:

> `useEmit` → channel routing подменяется через Provider в `startApp`-обвязке (`EmitProvider` принимает `eventSink` callback; в embedded — `eventSink = channel.send`; в standalone — `eventSink = localBus.dispatch`). HCA-слои используют `useEmit` единообразно. Реализация Provider'а — Phase 1a, **зависимость в `owner-web-core`** (рядом с `createCapsuleApp` из (a)). В Phase 1 demo-`hello` шлёт через `channel.send` напрямую — минимальный module, не полноценный HCA-app.

И в risks добавить пункт 7b:

> 7b. **`EmitProvider` (`useEmit` channel routing).** Без него `<Remote onClicked={cb}>` работает только с module'ами явно вызывающими `channel.send('clicked', ...)`. Phase 1a в зоне `owner-web-core`, рядом с (7a) `createCapsuleApp`.

## i) Serialization boundary — structured-clone, callback'и через props не передаются

postMessage клонирует через structuredClone. Объект-snapshot из `__capsule_remote_props__` envelope'а проходит structured-clone alg: примитивы / plain objects / Date / RegExp / TypedArray / Map / Set — переживают; **функции, Symbols, DOM nodes, class instances с приватными полями — НЕТ**.

Module-author напишет `<Remote computed={() => x()} schema={signal()}>` и:
- `signal()` — клонируется как plain object (ок).
- `() => x()` — silent dropped, callback теряется без warning.

Решение 5 (`on*` props) — canonical путь для callback'ов. Но это нужно явно сказать, иначе module-author попробует прокинуть helper-функцию через регулярный prop.

**Рекомендация:** добавить в «Известные свойства» пункт 9:

> 9. **Serialization boundary — structured-clone.** `__capsule_remote_props__` / `__capsule_remote_config__` envelope'ы клонируются через `postMessage`. **Callback'и (функции) через props не передаются** — silent dropping. Canonical путь для callback'ов = `on*` props (Решение 5). Class instances с приватными полями, Symbols, DOM nodes — также blocked. Документировать в AI-anchor + (в Phase 4) проверять в manifest schema validation.

## j) Reactive proxy — гарантия только на direct property access

Решение 4 описывает proxy-accessor поверх Solid store. `props.schema` корректно трекается Solid'ом — `get` trap читает `propsStore['schema']`, который реактивен. Но:

- `Object.keys(props)` через `ownKeys` trap отдаёт keys, **не триггерит subscribe на их значения**.
- `for (const k in props)` — то же самое.
- Spread `{ ...props }` копирует snapshot **на момент вызова**, далее не реагирует.
- `JSON.stringify(props)` — снимает snapshot.

Module-author напишет `for (const [k, v] of Object.entries(props))` для динамического рендера и удивится «почему не перерисовывается».

**Рекомендация:** в Решении 4 после описания `propsProxy` добавить:

> Reactive tracking гарантируется только на **direct property access** (`props.schemaName`). Enumeration (`Object.keys` / `for...in` / `{...props}`) возвращает snapshot на момент вызова; для динамической итерации module должен явно `createMemo(() => Object.keys(propsStore))`. Документировать в AI-anchor + примере bootstrap.

## k) Reserved event-name namespace `__capsule_*`

Iframe-shell слушает на `__capsule_remote_props__`, `__capsule_remote_config__`, `__capsule_remote_ready__`, `__capsule_remote_props_response__` (последние из brief'а). Если module через `channel.on('__capsule_remote_props__', cb)` подпишется руками — collision с shell'овым handler'ом, undefined behavior.

**Рекомендация:** добавить одну строку в Решение 6 (после reserved props table) или в Решение 2:

> **Reserved event-name namespace.** Префикс `__capsule_*` зарезервирован за shell-internal envelope'ами. `channel.on('__capsule_*', ...)` / `channel.send('__capsule_*', ...)` из пользовательского кода — runtime warning + no-op. Документировать в AI-anchor.

## l) `children` пока не поддерживаются

Решение 6 reserved-props таблица перечисляет четыре класса: System / Config / Events / Runtime props. `children` не упоминается. Composition внутри remote (`<Remote name="x"><span>...</span></Remote>`) требует slot-bridge / portal-across-frame механизма — это **отдельный архитектурный вопрос** уровня DnD-через-iframe (Risks#3).

**Рекомендация:** добавить пятый класс в таблицу Решения 6 или строкой ниже:

> `children` (JSX-композиция внутрь remote) — **не поддерживается в Phase 1**. Composition through frame boundary — отдельный ADR (slot-bridge / portal-across-frame), мотивируется по мере появления use-case'ов. В Phase 1 — TypeScript-уровневый ban (`<Remote>` не имеет `children?: JSX.Element` в props).

## m) Demo check #5 уточнение + минорные acceptance дополнения

(f) фиксирует `instanceId` requirement; добавлю что check #5 валидируется через **manual merge** в `startApp` (до landing'а `useAppConfig({ override })`).

**Рекомендация:** в Решении 7 в формулировке check #5 добавить parenthetical:

> Два instance'а ... — каждый видит свой `apiUrl` через `useAppConfig()` (в Phase 1 — через **manual merge внутри `startApp`** из `configOverride` параметра bootstrap'а; canonical `useAppConfig({ override })` API — Phase 1a от `owner-web-query`).

Минорные acceptance дополнения (1-2 строки каждое):

- **`openStandalone()` no-op gate.** В Phase 1 Acceptance gates: `useRemote().remote('hello').openStandalone({})` — не throw'ает, логирует warning, возвращает undefined. Это страховка от случайного `throw new Error('Phase 2')`.
- **`config={undefined}` semantics.** В Решении 3 явно: `<Remote config={undefined}>` эквивалентно отсутствию prop'а (provider+module merge применяется). Не интерпретировать как «обнулить всё» — иначе consumer случайно сотрёт ambient config.
- **`on[A-Z]` regex — non-collision с regular props.** В Решении 5 после регекса один комментарий: `online`, `onclick` (всё lowercase) НЕ matches `^on[A-Z]` — collision с legitimate props невозможна. Это уже implicit в коде, но явно прописать защищает от спекуляции «а вдруг `online`?».
