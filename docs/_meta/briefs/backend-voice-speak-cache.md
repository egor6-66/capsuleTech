# Контекст

`/voice/speak` — детерминированный GET: одинаковые `(engine, text, lang, voice, speed)` дают тот же WAV. Сейчас ответ без кэш-заголовков → браузер перекачивает одно и то же слово на каждый клик, и каждый запрос заново гоняет синтез (chatterbox ~8с CPU). ADR 054 закладывал server-side cache как штатный паттерн.

# Scope

**1. HTTP-кэш (браузерный ярус):**
- На успешный `/voice/speak` ответ: `Cache-Control: public, max-age=86400` + `ETag`.
- `ETag` = хэш **параметров запроса** (sha256 от канонической строки `engine|lang|voice|speed|text`), НЕ байтов — синтез детерминирован, хэшировать нечего дожидаться. ⚠️ `engine` в ключе — **resolved** (после подстановки дефолта из settings), не сырой query-параметр: смена `VOICE_ENGINE` не должна отдавать чужой кэш.
- Обработать `If-None-Match` → `304` **ДО** вызова движка (ревалидация не должна стоить синтеза).

**2. Серверный LRU-кэш синтеза:**
- In-memory LRU (например `cachetools.LRUCache`, maxsize по количеству записей ~512; WAV ~60KB → ~30MB потолок), ключ = тот же канонический хэш.
- Смотреть кэш до `eng.synthesize(...)`; класть после. Ошибки синтеза НЕ кэшировать.
- Диск-кэш/переживание рестарта — НЕ сейчас (отдельный шаг, если понадобится).

# Тесты

- Повторный `speak` с теми же параметрами → движок вызван ОДИН раз (fake-engine со счётчиком, как в существующих тестах).
- `If-None-Match` с верным ETag → 304, движок не вызван.
- Разный `engine`/`speed` → разные ETag, движок вызван снова.
- Заголовки `Cache-Control`/`ETag` присутствуют на 200.

# Acceptance

`uv run pytest` зелёные, `uv run ruff check .` — 0. Живой чек (может сделать user в браузере): два клика 🔊 по одному слову → второй запрос из disk/memory cache браузера (или 304), в логах voice синтез один.

# Что НЕ делаем

- Диск-кэш, pre-warm, streaming — нет.
- Фронт/learn не трогаем (кэш прозрачен для них).
