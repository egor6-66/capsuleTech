# web-remote — пересмотр направления (message-only / iframe)

> [!note] Статус: **обсуждение, решение не принято.** Заметка фиксирует разбор
> 2026-06-23, чтобы продолжить позже. Ничего в коде не менялось этим документом.

## Где стоим (факты)

- **ADR 056** (MF2 + iframe) — отвергнут на Шаге 0: `@module-federation/vite` proxy не
  ре-экспортит полный surface `solid-js/web` (захардкожен под React) → `SyntaxError:
  ... setStyleProperty`. Не чинится конфигом.
- **ADR 057** (native import-maps + shared Solid + shadow-DOM default) — принят и
  **смержен в main** (PR #420), но **в реальном браузере не тестился**. 86 unit-тестов
  (jsdom) зелёные, но jsdom не умеет `import('http://...')` / shadow mount → «работает ли»
  = неизвестно.
- В main висит мелкая TS-ошибка (каст в `importMap.test.ts:203-213` vs Vite-8 `ObjectHook`)
  → CI Typecheck красный. `Typecheck` **не** required-проверка, потому PR и утёк.

## Ключевой разбор (инсайт обсуждения)

### Два субстрата — это противоположные trade-off'ы

| | shadow-DOM (дефолт 057) | iframe |
|---|---|---|
| JS-реалм | **общий** с хостом | свой |
| `window`/`location`/History | **общие** → один роутер на всех | свои → своя навигация |
| Solid-инстанс | **один** (import-map) | свой |
| CSS | изолировать руками (shadow root) | изолирован сам |
| Связь | прямые ссылки в памяти | postMessage-канал |

### Диагноз бага «редирект на `/` (localhost:3050)»

`apps/universal-canvas` — это **полный app**: `remote.ts` зовёт `createCapsuleApp(root,
{ routeTree, … })`, т.е. поднимает целый роутер. Текущий дефолт-путь — **shadow-DOM
(общий реалм)**, выбран автоматически по origin (`LocalShadowDomTransport.canReach`
= same-origin). Значит **два роутера дерутся за один `window.location`** → роутер канваса
дёргает навигацию → URL хоста улетает в `/`. Не «не залогинен на :3000» (отдельной
страницы нет вообще), а конфликт роутеров в общем окне. Мульти-Solid даёт тот же
симптом — это баг именно shared-режима (в iframe его не бывает).

### Архитектурный баг текущего дизайна

Субстрат выбирается **по origin** (`canReach`), а не **по намерению**. Same-origin полный
app принудительно едет по shared-пути → редирект. Ручка `isolation="iframe"` помечена
Phase 2 → сказать «изолируй этот remote» сегодня нельзя.

**Нужно:** изоляция — явный режим в конфиге/пропсах, ортогональный origin'у:
- `mode: 'component'` (shared realm) — для встраивания-как-компонент.
- `mode: 'app'` (iframe) — для изолированного app внутри хоста.
- `standalone` — app сам по себе.

## Два реальных кейса (от пользователя)

1. **Карта мира — «компонент».** Standalone + внутри хоста сливается как компонент,
   берёт `apiUrl` из хоста. Самодостаточный виджет со своими стилями.
2. **Канвас в studio — «изолированный app».** Standalone + внутри хоста работает как
   полный app со своими глобалами/роутером, **не сливаясь** с хостом.

Оба ложатся на ось `component`/`app`. Это ровно замысел ADR 057 D4/D5 + ADR 053.

## Пересмотр: нужен ли shared-реалм вообще? (главный открытый вопрос)

Инсайт: **live-props-по-ссылке — единственная причина** существования shared-реалма и
всей import-map машинерии. Но **реактивные props через сообщения уже работают** (iframe-путь:
`__capsule_remote_props__` → remote `createStore` → effect'ы реагируют). Значит
shared-realm live-by-reference — **второй механизм на ту же задачу**, хрупкий (канон §0).

**Кандидат-решение:** схлопнуть в **один контракт (сообщения) поверх одного транспорта
(postMessage/iframe)**, import-map положить на полку.
- Различать: **транспорт** (как байты едут — деталь за `channel.send/on`; реально нужен
  один, postMessage; broadcast/socket/local-shadow = YAGNI) vs **контракт** (live-ссылки vs
  сообщения — схлопнуть в сообщения).
- Это **разворот ADR 057** (его тезис — import-map+shared Solid) → отдельный ADR-уровень
  (амендмент/supersede), принять осознанно.

**Что даёт:** убивает разом мульти-Solid, import-map, `/_shared/`, dedupe-конфликт,
shadow-Tailwind, и сам редирект (у iframe своя навигация — исчезает по построению).

**Что теряем (и важно ли для кейсов):**
1. Визуальное слияние с DOM/CSS хоста (inline) — iframe это прямоугольник со своим
   документом. Обоим кейсам не нужно (оба самодостаточны) → ок.
2. **DnD через границу** — pointer-события не пересекают iframe. **Единственная реальная
   заноза:** studio-палитра → drop в канвас. Если канвас в iframe — ломается.
3. Оверхед многих iframe / a11y / фокус через границы кадра — для пары крупных ремоутов
   не проблема.

## ⛳ Открытые вопросы (решить до фиксации ADR)

1. **В studio есть DnD из хоста в канвас** (палитра → drop)? Если да — iframe это ломает,
   нужен либо отдельный DnD-ADR, либо shared-режим именно для канваса.
2. Есть ли **хоть один** кейс, где remote должен **визуально слиться** с вёрсткой/CSS
   хоста (inline в форме хоста), а не быть самодостаточным прямоугольником? Если нет —
   message-only/iframe покрывает 100%, shared-реалм не строим.

## Предлагаемый план (черновик, после ответов на вопросы)

Сквозное правило: **одна переменная на шаг, один owner на пакет, gate «застабилено»
перед следующим шагом.** (Против «агенты навалили всё сразу».)

**Фаза 0 — навести порядок:**
- Починить тест-каст `importMap.test.ts` (owner-builders) + сделать `Typecheck`
  обязательной проверкой в branch protection (architect).
- Решить дубль `resolve.dedupe` vs `optimizeDeps.exclude` (не сносить вслепую).
- Сделать import-map громким в dev (что запинено / что скипнуто).
- Если идём message-only — определить scope «положить import-map на полку» (ADR).
- Минимальные фикстуры host/remote под выбранный первый кейс.

**Итерации (от простого, каждая со своим gate «застабилено»):**
1. Просто подтянуть модуль в хост (mount).
2. Научить общаться (props host→remote, события remote→host).
3. Изолированный app («мод для скоупа»).
4+ CSS-изоляция, multi-expose, prod/CDN.
