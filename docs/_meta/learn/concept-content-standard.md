# Профиль: КОНЦЕПТ (type: concept) — структура тела

> [!info] Frontmatter концепта — в [[authoring-contract-for-teacher]] (`type: concept`, `principle`, `kind: approach|pattern|recommendation`, `order`, `relatedRules`). Здесь — СТРУКТУРА ТЕЛА (контракт оставляет её как «проза»). Наследует [[core-content-standard]].

Концепт = **«как думать»** (оптика / ментальная модель), а не свод правил. Его задача — переустановить восприятие, поэтому он всегда доказывается контрастом и заканчивается процедурой.

## Скелет тела
`H1` → интро-`[!info]` (1–2 строки: суть подхода) → **PRINCIPLE** → тело-примеры (**RULE / CONTRAST / MATRIX** по нужде) → **«Как применять»** (STEPS) → (опц.) **NOTE** → (опц.) **MEDIA**.

## Обязательные блоки
1. **PRINCIPLE** — одна жирная строка-стержень (разворачивает frontmatter-поле `principle`).
2. **Контраст-доказательство** — хотя бы один блок «старая оптика ломается vs новая встаёт» (CONTRAST-таблица или пара RULE). Без него концепт голословен.
3. **«Как применять»** — STEPS (≥1 шаг). Без процедуры концепт не операционализирован.

## Специфика
Концепт всегда даёт три вещи: **(а)** образ/принцип, **(б)** контраст-доказательство, **(в)** процедуру применения.

Эталон — `word-as-image` («Слово — это образ, не перевод»):
- PRINCIPLE: учи образ и роль, не пару слово-перевод;
- контраст `face` (образ «лицом к» покрывает и сущ., и глагол — перевод ломается);
- контраст `condition` vs `circumstances` (перевод сливает синонимы, образ разводит);
- «Как применять» — 3 шага (спрашивай КАРТИНКУ; разные образы на путаемые пары; проверяй контекст образом).

## Профиль-чек-лист валидатора
1. PRINCIPLE присутствует и согласован с frontmatter `principle`.
2. Есть хотя бы один контраст-блок.
3. Есть «Как применять» (STEPS).
4. + весь core-чек-лист [[core-content-standard]].
