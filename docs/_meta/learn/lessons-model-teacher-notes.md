# Lessons: заметки учителя к модели (на обсуждение)

> [!abstract] Ответ на [[platform-context-for-teacher]]. Модель «либы + путь» из
> vault-брифа `brief-lessons-section` архитектором принята. Ниже — что ложится
> идеально, три места, где я бы докрутил, и вопросы на решение. Обсудить с Егором
> → потом финальный бриф с полями.

## Что ложится идеально (подтверждаю)
- **Ссылайся, не копируй** + **слово/смысл = join-key** — совпадает с замыслом vault.
  `grammar/` переиспользуем как «Правила» без копий, Lesson — сшивка.
- **Дрилл = данные (фаза 1), интерактив = код (фаза 2).** Контент не знает, КАК его
  проверяют. Правильное разделение.
- **Два `graboTag`** (классификатор в контенте vs личная история у аккаунта) —
  чёткая граница, согласен.

## Докрутка 1 — sense-based словарь усиливает «слово = образ», но нужен мост

Единица = **смысл** (bank₁/bank₂ — две карточки) — отлично. Но всплывает риск.

**Кейс `face`:** это теперь **два смысла** — `face` (лицо, сущ.) и `to face`
(противостоять, глаг.). По логике senses — две карточки. Опасность: они разъедутся
в два оторванных перевода, а это ровно то, с чем борется принцип «учи образ, не
перевод» (vault: `word-as-image.md`).

**Предложение (данные):**
- поле **`image` (образ)** на смысле — не перевод, а картинка-смысл;
- родственные смыслы связать **`related` / `derived_from`**;
- в UI показывать **общий образ-мост** между связанными смыслами
  (`face`: «повернуться ЛИЦОМ к чему-то» связывает и лицо, и «противостоять»).

Тогда `word-as-image` перестаёт быть методичкой и становится **данными**:
образ живёт на смысле, мост — на связи. Риск разъезда закрыт.

> Пример пары, где это критично: `condition` (требование/состояние) vs
> `circumstances` (ситуация вокруг) — по переводу слипаются, по образу разводятся.
> Это senses с разными `image`, а не один перевод «условия».

## Докрутка 2 — форма дрилла и «допуски» (ваш открытый вопрос) ⭐ главное

Из практики занятий вылезла жёсткая штука: **RU-промпт часто неоднозначен по
времени.** «Я пил кофе» = process now *или* habit → два разных ВЕРНЫХ EN. Значит
дрилл-данные обязаны это разруливать, иначе «почти верно» превращается в спор.

**Предлагаемый скелет Drill:**
```
Drill {
  promptRu,                    // "Я уже поел, когда он позвонил."
  context?,                    // опц. мини-ситуация (режим B) — снимает неоднозначность
  answerEn,                    // канон: "I had already eaten when he called."
  accept[],                    // равноверные варианты: ["I'd already eaten when he called"]
  nearMiss[ {pattern, hint} ], // "had ... call" → "точку даёт Past Simple, не had"
  graboTag                     // классификатор: "past-perfect-which-clause"
}
```

**Канон авторства дрилла (важно для importer'а):**
1. Промпт обязан быть однозначным: **либо маркер в тексте** (режим A: «вчера»,
   «уже», «сейчас»), **либо `context`** (режим B: мини-ситуация). Без этого дрилл
   невалиден.
2. `accept[]` — список равноверных ответов (стяжения, синонимичные порядки).
3. `nearMiss[]` — заготовленный точечный фидбек под типовую ошибку (то, что учитель
   говорит руками: «точку даёт Past Simple»). Это и есть ценность контента.
4. `graboTag` — только классификатор ошибки, без личной истории.

Связь с vault-каноном: «механика vs выбор времени — два режима», см.
`memory/drill-tense-markers`.

## Докрутка 3 — нейминг = id (принимаю, предлагаю зафиксировать сразу)

Раз имя файла = id и переименование = разрыв ссылок — фиксируем конвенцию до
первого импорта:
- папки: `lessons/`, `lessons/concepts/`, дриллы — `drills/` **или** `methods/drills/`;
- имена файлов — **kebab, тема-первой, сразу постоянные**:
  `past-perfect-which-clause.md`, `word-as-image.md`, `do-vs-be-questions.md`;
- никаких «temp/wip» в именах — id навсегда.

## Вопросы на решение (Егор + архитектор)

   принимаем как основу? От него зависит importer.
2. **Образ-мост между смыслами** — заводим поля `image` + `related/derived_from` на

4. Уровень L0–L5 на **Lesson** — подтверждаем, что рукой и отдельно от уровня слов.
