# Дополнения к стандарту контента — что заложить сейчас

> [!info] Четыре вещи, которых в стандарте пока нет, но которые дешевле заложить сразу, чем переписывать контент потом. По каждой: **что · зачем · предлагаемая форма · решение за вами.** Что примете — впишу в [[core-content-standard]].

## 1. MEDIA — якоря озвучки / образа
**Что:** формат ссылки-якоря на аудио (🔊) и картинку-образ прямо в теле примера.
**Зачем:** голосовому модулю нужен якорь произношения на каждый звук/пример; «слово = образ» просит картинку. В фонетике уже висит `todo: аудио-якоря` — без формата примеры придётся переписывать.
**Предлагаемая форма (тело автор не рисует UI, только помечает единицу):**
```
- go /ɡoʊ/ — идти            ← 🔊 и образ движок вешает по id единицы автоматически
```
То есть **автор ничего не проставляет руками** — озвучка/образ цепляются к слову по его id в словаре (где уже есть `pron_ru`/`ipa`/`image`). MEDIA-блок нужен только там, где якорь **не** привязан к слову (общий аудио-пример фразы) — тогда явный маркер. **Решение:** нужен ли явный MEDIA-маркер, или всё вешаем авто-по-id?

## 2. Провенанс для модерации
**Что:** поля в шапке контента — `author`, `updated` (дата), `reviewed_by`.
**Зачем:** контент правят модераторы — надо видеть кто/когда трогал и что отревьюено. Плюс отделяет **кураторское** от **авто-обогащения** (та же граница, что в словаре: ipa/frequency — авто, нюансы — руками).
**Предлагаемая форма (frontmatter, опц. но рекомендованные):**
```yaml
author: <кто создал>
updated: 2026-07-06        # ISO-дата последней правки
reviewed_by: <кто принял>  # пусто = не отревьюено
```
**Решение:** делаем обязательными для модерируемого контента или оставляем опциональными?

## 3. Версионирование контента
**Что:** поле `updated`/`version` + правило «менял `prerequisites`/структуру → бампни».
**Зачем:** когда у правила/концепта меняются связи или структура, вниз по цепочке (уроки, прогресс ученика) можно «осиротеть» — валидатору/миграции нужна зацепка, что изменилось. (Тот же риск флагали для словаря.)
**Предлагаемая форма:** хватит `updated` из п.2 как дешёвый вариант; полноценный `version: N` — если появится миграция прогресса.
**Решение:** `updated` достаточно на MVP, или сразу `version`?

## 4. Эмодзи-как-смысл — текст-эквивалент
**Что:** если эмодзи несёт информацию (🔊 «озвучить», ✅ «верно», ⚠ «осторожно»), рядом нужен смысловой текст/атрибут, а не только глиф.
**Зачем:** доступность (скринридеры) + чтобы валидатор не считал значимый эмодзи «украшением» и не резал.
**Предлагаемая форма:** эмодзи-декор в тексте — свободно; эмодзи-**смысл** — только через семантику движка (иконка кнопки/вердикта рисуется китом, у неё есть текст-лейбл), в сыром теле смысл эмодзи не кодируем.
**Решение:** зафиксировать «в теле — только декоративные эмодзи; смысловые = через компоненты кита»?
