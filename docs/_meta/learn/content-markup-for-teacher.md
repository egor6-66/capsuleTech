# Стандарты оформления контента → под них валидаторы (нужен твой взгляд)

> [!info] Зачем
> Скоро контент правят **модераторы** — и не только learn (доки, темы, справочники, любой
> текст). Чтобы всё было в **одной стилистике** и проверяемо, нам нужны **жёсткие стандарты
> оформления**: «один тип контента → один тип блока → один формат». Ты придумываешь **форму**
> (какая разметка для какого блока/темы), мы под неё пишем **валидаторы** (скрипт режет всё,
> что не по стандарту).
>
> **Про технику не думай.** Механику разметки (якоря заголовков, обёртки, стили) накидываем мы
> под капотом. Ты описываешь **семантику и формат** — что за блок, в каком порядке, как выглядит.

## Прецедент (это уже работает — делаем по образцу)
Ты уже написал **`rule-content-standard.md`** — эталон для rule-доков: скелет документа
(frontmatter → H1 → интро-`[!info]` → блоки → todo), **закрытый список блоков** (RULE / CONTRAST /
MATRIX / SOUND / NOTE), строгий формат строки-примера, чек-лист валидатора. **Это идеальный
шаблон.** Просьба — сделать **то же самое для остальных типов контента**.

## Вокабуляр разметки, который наш движок рисует и стилизует
(markdown → `renderMarkdown` → `Prose`, всё на design-токенах, тёмная тема авто). Кирпичики:
- **Frontmatter** (YAML сверху) — `tags`, `type`, `status`, `order`, и т.п. Обяз. поля = валидируем.
- **Заголовки** `#`…`######` — H1 = заголовок сущности (ровно один), H2/H3 = секции/подблоки.
- **Callout'ы** `> [!info]` / `[!note]` / `[!warning]` / `[!todo]` — цветные блоки-плашки.
  (набор типов = то, что стилизует наш рендер; нужен ещё тип — заведём).
- **Таблицы** — параллельные данные (2+ колонки, шапка). Стилизуются как грид.
- **Списки** — буллеты/нумерация. Плоские иллюстрации: один пункт = одна единица.
- **Код** ``` ``…`` ``` — моноширинный (для форм/паттернов/IPA-наборов).
- **Жирный/курсив** — акцент (напр. принцип правила = **жирная строка**).
- **Wikilinks** `[[ref]]` — ссылка на другой док/сущность (кликабельна, ведёт внутри платформы).
- **Конвенции формата** (из rule-стандарта): разделитель нескольких единиц — ` · ` (middot),
  тире `—` перед переводом, IPA в слэшах `/…/`.

Из этого набора собирается любой блок. Скажи какой блок под какой контент — соберём + провалидируем.

## Что просим спроектировать (по образцу rule-стандарта)
Для КАЖДОГО типа контента, который правят модераторы, — свой мини-стандарт той же формы:
1. **Скелет** — жёсткий порядок частей документа.
2. **Frontmatter-схема** — какие поля обязательны/закрытые списки.
3. **Закрытый список блоков** — какие секции разрешены, шаблон каждого (строго).
4. **Форматы строк** — машинно-проверяемые (как строка-пример в rule-стандарте).
5. **Правила анти-бардака** — таблица vs список, дедуп/маршрутизация, «нет текста вне блоков».

Типы контента на твоё усмотрение (накидал — дополни/переименуй):
- **Концепты** (`type: concept`) — статьи «как думать» (принцип + тело + примеры + связанные правила).
- **Уроки / тексты дриллов** — если у них своя разметка помимо rule.
- **Темы / разделы** (обзорные доки, оглавления разделов).
- **Общие доки платформы** — если хотим единую стилистику и для них.
- (rule уже готов — не трогаем, он образец.)

## Формат ответа
Как удобно — можно прямо файлами-стандартами (как `rule-content-standard.md`), можно наброском
«вот такие блоки, вот формат». Мы переведём каждый твой стандарт в **валидатор** (скрипт для
модераторов) + сможем **авто-накидывать** часть разметки под капотом. Цель — доки, темы и
learn-контент в одной стилистике, проверяемо.
