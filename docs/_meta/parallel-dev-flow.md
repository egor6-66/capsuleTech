# Parallel dev flow — `develop` trunk + path-scoped commits

> Действующая конвенция для **параллельной работы**: пользователь + несколько
> независимых агентов на **одном репозитории**, скоупы не пересекаются, и
> пользователь видит **живьём** на своём dev-сервере всё, что делают все агенты.
> Согласовано 2026-06-05. Это **не** замена per-package OWNERSHIP — это правила
> git/ветвления поверх него.

## ⭐ Скоуп = пакет/апп. Owner коммитит только свой скоуп (база, 2026-06-10)

**Скоуп — это КАЖДЫЙ пакет и КАЖДЫЙ апп по отдельности** (`packages/web/ui`,
`packages/web/style`, `packages/web/map`, `packages/web/table`, `apps/playground`,
…), а **НЕ «правки агента vs правки человека»**.

- **Owner работает и коммитит ТОЛЬКО свою зону.** Один коммит/PR = один скоуп со
  **ВСЕМИ** его изменениями (независимо от авторства строк). Scope-тег:
  `feat(web-ui): …`, `feat(web-style): …`.
- **НЕ дробить скоуп по авторству.** Если в пакете есть изменения и от агента, и от
  человека — это **одна единица работы пакета**, коммитится owner'ом целиком.
  Забрать «свои файлы» и оставить остальное **нельзя** → история рвётся, «половину
  потом не найти». (Стейдж всего скоупа: `git add packages/web/ui`, НЕ по-файлово.)
- **Агенты не лезут в чужой скоуп.** Нужна правка в соседнем пакете → не делаешь
  сам, **эскалация к главному (человеку)**, он зовёт нужного owner'а.
- **Мультискоуп-задача** → каждый owner делает СВОЮ зону, **отдельный PR на скоуп**;
  вся координация между скоупами идёт **через главного (человека)**. Не смешивать
  скоупы в одном PR.
- **Прецедент-ошибка (PR #284, 2026-06-10):** взял свои web-ui-props по-файлово,
  оставил select/textarea→input рефактор того же пакета + затащил web-auth
  `loginForm` в тот же PR → дробление web-ui по авторству + смешение двух скоупов.
  Правильно было: **web-ui PR** (props + select/textarea рефактор вместе, owner-web-ui)
  и **отдельный web-auth PR** (owner-web-auth).

## ⭐ Правки человека — часть общей работы, НЕ «WIP»

Работаем в **паре, по очереди**: пишет код агент — человек не лезет; агент
простаивает — человек вносит правки. Это **совместная работа**.

- Человек **может вносить правки в любом скоупе**. Это **НЕ чужой WIP**, который
  агент должен обходить/исключать из коммита.
- Если человек поправил что-то в зоне owner'а и **сломал / изменил нужное
  поведение** — owner **говорит главному**, разбираемся вместе.
- **Критичные** правки человек **сообщает сразу**. Правки уровня **текста /
  незначительных стилей** — можно не обсуждать.

## ⭐ Дев-сервер и браузер-верификация (2026-06-10)

Дев-серверы гоняет **человек (главный)**, агенты к ним **подключаются на чтение**, а НЕ поднимают свои.

- **Сервера в `.claude/launch.json`.** Текущий эталон — `playground-dev` на **порту 3050** (`apps/playground`). Человек держит его запущенным; **HMR работает** — uncommitted-правки в `packages/*` и `apps/*` доходят в запущенный апп живьём (НЕ из dist).
- **Не гадать «дошла правка или нет» — СМОТРЕТЬ.** Для верификации визуала/поведения использовать **Chrome MCP** (`mcp__Claude_in_Chrome__*`): `tabs_context_mcp` → `navigate` на URL (напр. `http://localhost:3050/login`) → `javascript_tool` с `getComputedStyle` для **точных замеров** (цвет бордера, gap, высота…). Замер > скриншот > догадка про HMR/dist/резолв.
- **`preview_start` (Claude Preview) НЕ цепляется к внешнему серверу человека** — порт занят его процессом, поднимать конкурирующий нельзя (и убивать его сервер — тоже). Поэтому путь — Chrome MCP к его запущенному серверу.
- **Ограничение автоматизации:** псевдоклассы `:focus` / `:focus-visible` в браузер-автоматизации **не матчатся** (нужен фокус OS-окна, которого при программном осмотре нет — все focus-тесты вернут false). → focus/ring-баги headless **не воспроизвести**; подтверждает их **человек реальной интеракцией**.
- Грабля: новый файл/экспорт/subpath пакета → Vite может зависнуть на reload (mid-rebuild) → нужен ЧИСТЫЙ рестарт dev + purge `apps/<app>/node_modules/.vite` (делает человек — сервер его).

## Зачем именно так

«Видеть на dev-сервере всё сразу» и «у каждого агента своя ветка» —
взаимоисключающие. Отдельная ветка/worktree = отдельное дерево файлов; пока не
смержишь — не видно. Значит **все агенты редактируют одно рабочее дерево**
(локальный чекаут пользователя, с которого крутятся серверы), а изоляция —
**по путям (scope = папка)**, не по веткам.

## Модель: trunk на `develop`, коммиты по путям

- **Одна ветка `develop`** (от свежего `main`). Dev-серверы пользователя крутятся
  с неё. Все агенты пишут в **этот же** чекаут → правки видны живьём (даже до
  коммита — та же ФС, HMR подхватывает).
- **Изоляция по scope-папке.** Один агент = один префикс пути. Пример текущей
  раскладки:
  | Агент | Scope-папка |
  |---|---|
  | главный (workspace) | `apps/testhub/**` |
  | owner-web-agent | `packages/web/agent/**` |
  | ewc-агент | `apps/ewc/**` |
- **`develop → main`** — на вехах, через PR (ревью + CI). Коммиты scope-тегированы
  и не пересекаются по путям → мердж чистый.

## Жёсткие правила (на этом обжигались — НЕ нарушать)

1. **Никогда `git add -A` / `git add .`.** Только явные пути своего scope. Дерево
   общее — стейджить чужое нельзя. (`git add apps/testhub/...`, не `git add -A`.)
2. **Scope-тег в commit message.** `feat(web-agent): …`, `feat(testhub): …`,
   `feat(ewc): …`. Один `git log` — видно, кто что сделал.
3. **Shared-infra — только главный.** `tsconfig.base.json`, `packages/builders/**`,
   `nx.json`, root `package.json`, `CLAUDE.md`, `docs/_meta/**`, `scripts/**`,
   `docker/**`. Главный их трогает и **объявляет**. Агенты — не редактируют,
   **запрашивают** у главного. Это главный источник коллизий между скоупами.
4. **Один файл — один scope.** Если два скоупа упёрлись в один файл → это
   нарушение границы: эскалация к главному/owner'у, **не правим оба**.
5. **Без topic-веток во время активной параллельной работы.** Они заставляют
   постоянно мержить и ломают «живой» вид. Topic-ветка — только под рискованный
   refactor, который scope хочет спрятать; мердж в `develop` по готовности.
6. **Push в remote — сериализованно**, делает главный (или `owner-git`). Локально
   `.git` общий — гонок нет; гонка только на `git push` (non-fast-forward).
7. **Worktree-изоляция субагентов — НЕ для «покажи живьём».** `isolation:
   "worktree"` кладёт работу в отдельную директорию → пользователь её не видит на
   своём сервере. Для видимой работы агенты пишут in-tree; worktree — только под
   одноразовую разведку.

## Почему это закрывает задачу

- **Видишь всё живьём** — один чекаут, dev-сервер показывает суммарную работу всех
  агентов мгновенно.
- **Не путаешься в скоупах** — папка = владелец, scope-тег в логе, `-A` запрещён →
  чужое не подмешивается.
- **Чисто доезжает в `main`** — непересекающиеся пути → беспроблемный мердж по вехам.
