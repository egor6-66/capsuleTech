# Как устроена платформа — контекст для учителя

> [!abstract] Это карта того, что в аппе УЖЕ есть, что планируется, и как контент
> из vault попадает на экран. Цель — чтобы бриф по [[brief-lessons-section]]
> опирался на существующие механики, а не переизобретал их.

## Что уже есть и работает

### 1. Словарь (библиотека слов) — сердце системы
- Единица словаря — **смысл** (sense), не слово: у «bank» два смысла = две
  отдельные карточки. Смысл несёт: оригинал, перевод (ru), фонетику (pron_ru),
  определение en (gloss), теги, уровень, связи с другими смыслами
  (синонимы/related — как synset happy/glad/joyful).
- **Уровни L0–L5** на каждом смысле — уже есть, твой teacher-vocab корпус
  (~170 смыслов) импортирован и живёт в разделе Library аппа.
- **Слово = join-key**: любой будущий контент (дриллы, примеры в правилах,
  песни) ссылается на слово/смысл, а не копирует его. Карточка слова, перевод
  и озвучка «приезжают» по ссылке автоматически.

### 2. Озвучка
Каждый смысл в выдаче уже несёт готовую ссылку на озвучку (🔊 в плитке и в
карточке). Несколько TTS-движков, переключаются в шапке аппа. **В уроках ничего
про озвучку описывать не надо** — где есть слово, там будет и звук.

### 3. Аккаунты
Вход/регистрация уже работают (единый аккаунт на всю экосистему). Это важно
для будущего: «личные грабли», прогресс, spaced-возврат — всё это привязывается
к аккаунту. Сейчас апп работает и без входа (гость видит контент).

## Как контент попадает в апп (главное!)

```
vault (Obsidian, ты пишешь)  →  importer (скрипт)  →  база  →  апп
```

- **Vault — место авторства.** Апп НИКОГДА не читает .md напрямую. Заметки
  проходят через importer, который парсит структуру и кладёт в базу с пометкой
  «curated» (учительский контент).
- Правки в vault → перезапуск импорта → апп видит новое. Не мгновенно, но
  дёшево.
- Следствие: у заметок должна быть **предсказуемая структура** (frontmatter-поля,
  стабильные имена/id). Конкретный формат полей согласуем, когда утвердим
  модель — сейчас важно просто знать это ограничение.

## Что планируется под Lessons (как ляжет твой бриф)

Твоя модель (либы + путь) принята, деление такое:

| Твоё | Что это технически | Когда |
|---|---|---|
| Концепты (проза) | заметки-статьи, апп рендерит markdown красиво — механика уже есть | фаза 1 |
| Правила (grammar/) | переиспользуем твой grammar/ как есть, НЕ копируем в уроки | фаза 1 |
| Дрилл (описание) | **данные**: prompt RU, эталон EN, допуски, тег грабли | фаза 1 (read-only) |
| Дрилл (интерактив) | **код**: поле ввода, проверка, фидбек — делает апп, не контент | фаза 2 |
| Личные грабли, spaced-возврат | прогресс юзера, живёт у аккаунта, НЕ в контенте | фаза 3 |
| Lesson (путь) | заметка-сшивка: ссылки на концепты+правила+дриллы (+ свой уровень) | фаза 1 |

> [!important] Два разных «graboTag»
> В контенте дрилла тег грабли = **классификатор** («тут типовая ошибка с
> do-support»). А «Егор наступил на эту граблю 3 раза, вернуть через неделю» —
> это **личная история**, другой слой (аккаунт), контент про неё не знает.
> В заметках описываем только классификатор.

## Правила авторства (чтобы контент лёг в модель без переделок)

1. **Ссылайся, не копируй.** Урок ссылается на правило/концепт/дрилл. Одно
   правило — в десятках уроков. Копия = рассинхрон навсегда.
2. **Слова из словаря — по слову.** В дрилле пишешь слово (bank) — апп сам
   подтянет смысл/перевод/озвучку. Если слова нет в словаре — добавляем словом
   в словарь, не описываем внутри урока.
3. **Стабильные имена заметок.** Имя файла/заметки станет id — переименование
   после импорта = разрыв ссылок. Выбирай имена сразу как постоянные.
4. **Уровень урока — отдельно от уровня слов.** Дрилл на словах L0 может быть
   уроком L3 (по концепту). Проставляй уровень урока рукой, из слов он не
   выводится.
5. **Интерактив описываем данными.** «Покажи RU, жди EN, прими вариант X и Y,
   при ошибке Z скажи про do-support» — это поля заметки-дрилла. КАК это
   выглядит и проверяется — забота аппа.
