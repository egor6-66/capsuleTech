# Owner-orchestration — как рулить овнерами параллельно

> Практический чек-лист: пользователь ведёт несколько owner-агентов на **одном
> репозитории**, не теряя их контекст и не давая правкам затирать друг друга.
> Слой **поверх** [`parallel-dev-flow.md`](parallel-dev-flow.md) (git-правила) и
> per-package `OWNERSHIP.md` (зоны). Согласовано 2026-06-09.

## 🔑 Главный принцип — изоляция по СКОУПУ, не по сессии

Все агенты пишут в **одно рабочее дерево**. Значит два агента, редактирующие
**один файл → last-write-wins (затирание)** — независимо от того, форк это или
отдельная сессия. **Форк сессии НЕ даёт файловой изоляции.**

> Параллелить можно только **РАЗНЫЕ скоупы** (папки) или непересекающиеся файлы.
> Один скоуп = задачи **последовательно**.

## ⚠️ Анти-паттерн (на чём ловятся)

«Дал овнерство агенту → он изучил пакет → форкаю его под каждую задачу» — форки
**внутри одного пакета**, трогающие одни файлы, **перезапишут друг друга**, и
каждый форк считает себя единственным владельцем. Не делать.

## ✅ Два режима работы

### Режим A — последовательно в одном скоупе (дефолт)
Держишь **ОДНУ живую сессию овнера** на пакет. Новую задачу даёшь **тем же
`SendMessage` (resume)**, а НЕ новым форком:
- овнер сохраняет весь контекст → ноль повторного чтения;
- ноль коллизий (он один в своём пакете);
- закончил задачу A → коммит (path-scoped) → задача B в той же сессии.

### Режим B — параллельно по РАЗНЫМ скоупам (для скорости)
Разные овнеры на разные пакеты одновременно — они физически не пересекаются по
файлам:
```
owner-web-creator (packages/web/creator)  ‖  owner-cli (packages/cli)  ‖  главный (apps/playground)
```
Каждый коммитит свой scope (scope-тег, явные пути).

### Параллель ВНУТРИ одного пакета — только если очень надо
Дели по **subpath** (один делает `/tree`, другой `/inspector` — разные файлы),
коммить по явным путям. Совсем изолированно — git **worktree** (отдельная папка),
но тогда теряешь «живой» вид на dev-сервере и надо мержить. Обычно проще режим A.

## 🧠 «Овнер не теряет контекст» — решается доком, не сессией

Долговечный контекст овнера — это **`OWNERSHIP.md`** (+ `docs/_meta/<pkg>.md`,
`docs/playground/`), а не память сессии. Любой свежий/форкнутый овнер **первым
делом читает `OWNERSHIP.md`** (зашито в его agent-определение) и сразу в теме.

> ⇒ Держи `OWNERSHIP.md` **актуальным** — овнер дописывает его по ходу. Тогда
> даже если сессия умерла/форкнулась, контекст восстанавливается за одно чтение.
> **Не полагайся на «живую» память длинной сессии** как на источник правды.

## 🧵 Швы между скоупами фиксирует главный (contracts-first)

Где скоупы встречаются (тип из `web-contract` → потребляет `web-creator`;
`Page.meta {auth,can}` → guard в web-router; метадата сборки CLI↔preview-server) —
**главный фиксирует контракт/тип первым**, потребитель **мокает** его и работает
параллельно, не дожидаясь и не клобберя продьюсера.

## 📋 Чек-лист на каждую задачу

1. **Один скоуп?** Да → режим A (resume овнера). Разные → режим B (параллельно).
2. **Пересекаются файлы с другой активной задачей?** Да → сериализуй или раздели по subpath.
3. **Шов с чужим скоупом?** Да → главный фиксирует контракт первым, ты мокаешь.
4. **Бриф овнеру self-contained?** (что + какие файлы + шов-контракт + ссылки на доки.)
5. **Коммит:** явные пути (НИКОГДА `git add -A`), scope-тег (`feat(creator): …`), часто.
6. **PR/push → owner-git** (сериализованно). После CI-green — мерж.
7. **Овнер обновил `OWNERSHIP.md`?** (если поменялись контракт/quirks/план.)

## 🔁 Поток одной задачи

```
главный: бриф + шов-контракт  →  овнер делает в своём скоупе  →  главный: ревью/тест  →  owner-git: PR → merge
```

## 🗺️ Текущая раскладка (playground-инициатива)

| Скоуп | Овнер | Режим |
|---|---|---|
| `packages/web/creator` | owner-web-creator (1 живая сессия) | A |
| `packages/web/contract`, `packages/web/access` | главный (стюард) | A |
| `packages/cli` | owner-cli | B (параллельно creator) |
| `apps/playground` | главный (playground owner) | B |
| прочие пакеты по мере (web-style/router/ui/...) | соответствующие owner'ы | B |

## 🚫 Красные флаги

- Два агента в одном файле → **стоп**, нарушена граница → к главному.
- Форк овнера под перекрывающуюся задачу → затирание → используй resume (режим A).
- `git add -A` на общем дереве → подмешивание чужого → только явные пути.
- Полагаться на память длинной сессии вместо `OWNERSHIP.md` → потеря контекста при форке/рестарте.
