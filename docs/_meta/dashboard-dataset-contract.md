# Дашборд: контракт данных для виджетов (таблицы + графики)

> Рекомендация фронт-архитектора для backend-разработки. Статус: предложение под обсуждение,
> при старте работ оформим ADR. Вопрос, на который отвечает док: «две структуры данных
> (таблицы/графики) или одна универсальная?»

## TL;DR

Ни то, ни другое. Бест-практис индустрии (Grafana data frames, Vega-Lite, ECharts dataset):
**один канонический формат данных + отдельные view-спеки как проекции**. Граница проходит не
«таблица vs график», а «данные vs представление».

- **Данные** — всегда один формат: **tidy/long** (плоские записи + мета полей). Бэкенд отдаёт
  только его.
- **Представление** — discriminated union по `view` (table | chart) на стороне фронта; график
  получает серии через декларативный encoding-маппинг, таблица ест rows напрямую.

## Формат данных (то, что отдаёт API)

```jsonc
{
  "fields": [
    { "key": "date",    "label": "Дата",    "type": "dimension", "format": "date" },
    { "key": "region",  "label": "Регион",  "type": "dimension" },
    { "key": "revenue", "label": "Выручка", "type": "measure", "unit": "RUB", "format": "money" },
    { "key": "orders",  "label": "Заказы",  "type": "measure", "format": "int" }
  ],
  "rows": [
    { "date": "2026-07-01", "region": "MSK", "revenue": 120500, "orders": 342 },
    { "date": "2026-07-01", "region": "SPB", "revenue":  80100, "orders": 210 },
    { "date": "2026-07-02", "region": "MSK", "revenue": 131700, "orders": 355 }
  ]
}
```

Правила:

1. **Tidy/long, не wide.** Одна запись = одно наблюдение. НЕ разворачивать измерение в колонки
   (`revenue_msk`, `revenue_spb` — нельзя): wide-формат необратим без знания схемы, long
   проецируется в любой pivot на клиенте.
2. **`fields` обязателен.** `type: dimension | measure` — рабочая лошадка: из него автоматически
   выводятся и колонки таблицы, и допустимые оси/серии/легенды графика. Плюс `format`/`unit` —
   форматирование в одном месте, фронт не гадает.
3. **Скаляры в rows.** `string | number | boolean | null`. Даты — ISO-строки. Без вложенных
   объектов: вложенность = сигнал, что это два датасета.
4. **Один формат для всех виджетов.** KPI-плитка — тот же датасет из 1 row / 1 measure.

## Почему НЕ два формата (отдельно «для таблиц» и «для графиков»)

- Дублирование запросов и агрегационной логики на бэке.
- Таблица и график **одних и тех же цифр** начинают дрейфовать (разные округления, фильтры,
  таймзоны) — классический источник «в таблице одно, на графике другое».
- Убивает типовую фичу дашборда — переключение таблица⇄график одним кликом: с единым датасетом
  это смена view-спеки на клиенте, с двумя форматами — второй эндпоинт и вторая ветка логики.

## Почему НЕ «одна универсальная структура под оба рендера»

Структура, которая пытается быть «сразу и пропсами таблицы, и пропсами графика», вырождается либо
в скрытый union (лже-универсальность), либо в компромиссы, портящие оба рендера. Универсальными
должны быть **данные**, а не виджет-пропсы. Пропсы конкретного рендера — это проекция:

```ts
// фронт, не бэкенд:
interface IWidgetSpec {
  query: IQueryRef;                  // один источник данных
  view: ITableView | IChartView;     // union по ПРЕДСТАВЛЕНИЮ, не по данным
}
// IChartView.encoding: { x: 'date', y: 'revenue', series: 'region' }  ← Vega-Lite/ECharts-стиль
```

## Агрегация — на сервере

Таблице нужны сырые/подробные строки (много, с пагинацией), графику — агрегат (мало точек).
Это могут быть **два запроса от одного query-описания** (разный `groupBy`/`granularity`), но
формат ответа всегда один и тот же tidy. Не гонять сырьё на клиент ради агрегации в JS.

Практические следствия для API:

- Эндпоинт принимает: метрики, измерения, фильтры, `granularity` (для дат), пагинацию/лимит.
- Отвечает всегда `{ fields, rows }` + пагинационная мета где применимо.
- Округления/таймзоны — на сервере, единообразно (см. пункт про дрейф).

## Прецеденты

- **Grafana data frames** — единый columnar-формат для всех панелей (таблицы, графики, KPI).
- **Vega-Lite** — `data` (tidy) + `encoding` (маппинг поле→канал).
- **ECharts** — `dataset` + `encode`, один датасет на несколько серий/видов.
- **Tidy data** (Wickham) — теоретическая база long-формата.

## Как это ляжет на фронт (справочно, capsule/HCA)

Entity = zod-схема датасета · Shape per-представление (таблица-Shape и чарт-Shape над одной
Entity, two-phase ADR 036) · Widget композирует. `Ui.Chart` и table-стек в ките уже есть —
контракт выше стыкуется без адаптеров.
