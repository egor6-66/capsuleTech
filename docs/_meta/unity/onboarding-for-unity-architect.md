# Онбординг: экосистема capsule и как в неё встраивается Unity

> [!abstract] Ты делаешь миры/механики в Unity. Мы — веб-экосистема с готовой
> инфраструктурой: аккаунты, контент-база для изучения языка, голосовые
> нейросервисы, рейтинги. Этот док — что уже есть, как собирать под нас, и
> как мы общаемся. Идеи интерактивов — [[../learn/unity-interactives-ideas]]
> (бриф учителя английского). Отвечай файлами в этой же папке.

## Что за система (2 минуты)

- **capsule** — веб-экосистема на своём фреймворке: несколько приложений
  (изучение английского, комьюнити-соцсеть, студия) на ОДНОМ домене за
  nginx-gateway; все запросы к бэкам — `https://<домен>/api/<сервис>/...`.
- **Аккаунты готовы**: вход по httpOnly-куке; в браузере она едет с любым
  same-origin запросом автоматически (и из WebGL-билда тоже).
- **Контент-база (lang)**: слова со смыслами/переводами/фонетикой/уровнями
  L0-L5, правила грамматики, дриллы с готовым каталогом типовых ошибок
  (nearMiss: «как ошибаются и что подсказать»). Всё отдаётся по HTTP API.
- **Нейросервисы self-hosted**: TTS-озвучка (готово), генерация картинок
  (готово), LLM-генерация со structured-JSON (готово), STT-распознавание —
  в ближайшей очереди. Все — HTTP API за тем же gateway.
- **Рейтинги/прогресс**: журнал событий в комьюнити — игра шлёт «сессия
  завершена, score X» → очки/лидерборды считаются у нас.

## Как Unity встраивается (архитектура — НЕ обсуждается, канон)

1. **Билд-таргет: WebGL.** Мир грузится ВНУТРИ веб-приложения как удалённый
   модуль (наша remote-механика; прецедент — 3D-канвас уже так живёт).
   Никаких standalone-лончеров на старте.
2. **Мост Unity ↔ веб-хост** — типизированные JSON-сообщения через
   postMessage/jslib (контракт версионируется файлом, мы его вендорим на
   обе стороны — механика обкатана). Хост шлёт: init-конфиг (юзер, тема,
   уровень, фраз-сеты), данные по запросу. Unity шлёт события: выбор
   подсказки, старт/стоп записи голоса, завершение сцены, телеметрию.
3. **Все внешние вызовы билда — ТОЛЬКО на наш origin** (`/api/...`).
   Хардкод чужих URL/сервисов в билде запрещён (у нас это канон для всех
   узлов системы). Конфиг приходит через init-мост, не зашивается.
4. **Контент не дублируется.** Слова/фразы/сценарии приезжают из наших API
   по ссылке. В билде — только механики, модели, сцены. Игра = рендер и
   механика поверх наших данных.
5. **Ассеты** — бандлы/Addressables, хостятся у нас как статика (отдадим
   раздел за gateway). Бюджет размера обсуждаем, ориентир первого мира:
   initial-загрузка ощутимо меньше 100 МБ (Brotli).

## Голос (для ролеплей-режима)

Микрофон-капчер на твоей стороне (WebGL microphone API через jslib), голос —
чанками на наш STT-endpoint; ответы NPC — готовые аудио-URL с нашего TTS.
Логика «кто что говорит и как оценивается» — наш бэк; ты рендеришь и
проигрываешь.

## Онлайн — поэтапно (важно: не строим лаг-критичное раньше времени)

- **Фаза 0 — БЕЗ сети**: соло-ролеплей с NPC-нейронкой (наш STT+LLM+TTS).
  Первая цель.
- **Фаза 1 — turn-based 1-на-1**: собеседники говорят ПО ОЧЕРЕДИ (push-to-
  talk) → латентность некритична → транспорт = наш WebSocket-канал (готовится
  в комьюнити). Никакого игрового netcode.
- **Фаза 2 — лобби/комнаты/темы/рейтинг** поверх того же канала.
- **Фаза 3 — free-roam/шутер**: настоящий state-sync. ТУТ нужна твоя
  экспертиза — вопросы ниже. Ограничение: только self-hosted решения
  (Unity Gaming Services/Relay и прочие внешние облака НЕ подходят —
  системный канон; сервер дадим наш).

## Как мы общаемся

Как с учителем английского: **файлы в `docs/_meta/unity/`**, передаёт Егор.
Твои ответы/предложения/вопросы — отдельными файлами со статусом в
frontmatter (`proposal | question | done`). Решения по интеграции фиксирует
architect (я) в ADR-доках; тебе их перескажу выжимкой сюда же.

## Первые вопросы к тебе

1. **Версия Unity и шаблон проекта** под WebGL (URP? билд-сайз опыт?).
2. **Netcode для фазы 3, self-hosted**: что предлагаешь (Mirror / FishNet /
   Netcode for GameObjects с кастомным transport'ом на наш WS/UDP-relay)?
   С пониманием: фазы 0-2 без него.
3. **Пайплайн 3D-ассетов** (вопрос учителя): библиотека объектов заранее +
   догрузка бандлами, или генерация? Мы умеем генерить 2D-картинки из
   текста-образа; 3D — твоя территория, предложи реалистичный путь.
4. Микрофон в WebGL: подтверди рабочий путь капчера (getUserMedia через
   jslib) и формат чанков, удобный тебе для отдачи нам.
5. Что тебе нужно от нас в первую очередь для прототипа «sandbox предлогов»
   или «соло-ролеплей» (это два кандидата на первый мир — см. идеи учителя)?
