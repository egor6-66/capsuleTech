# Ответ на онбординг: стек, netcode, ассеты, микрофон, первый мир

> [!info] Канон интеграции принят без возражений: WebGL-модуль внутри веб-хоста,
> мост через postMessage/jslib (версионируемый контракт), все вызовы только на ваш
> origin, контент по ссылке, ассеты — Addressables у вас как статика. Ниже — по
> пунктам.

## 1. Версия Unity + шаблон под WebGL
- **Unity 6 LTS (6000.x).** Лучший на сегодня WebGL: меньше билд, WebAssembly 2023,
  pthreads (SharedArrayBuffer), улучшенная память/GC, поддержка сжатия текстур.
  2022.3 LTS — запасной вариант, если команде важнее «проверенное», но новый проект
  я бы стартовал на 6.
- **Render pipeline: URP в «мобильном» профиле.** WebGL по бюджету ≈ мобилка:
  renderer-asset без тяжёлого пост-процесса, baked-lighting где можно, ограниченные
  realtime-тени. URP даёт Shader Graph и масштабируемость; Built-in чуть легче на
  совсем пустом билде, но URP окупается стилизованным low-poly.
- **Размер билда:** фикс-косты движка+кода ~10-20 МБ (Brotli) — это основная масса
  для «sandbox»; остальное режем: IL2CPP + **managed stripping High**, выключить
  неиспользуемые модули (Physics2D, Terrain, Video…), Crunch/ASTC-текстуры,
  **аудио НЕ бандлим** (TTS-озвучка едет URL-ами с вашего бэка). <100 МБ — с большим
  запасом для первого мира.
- ⚠️ Для pthreads/AudioWorklet нужен **cross-origin isolation** (заголовки
  `COOP: same-origin` + `COEP: require-corp`) на странице-хосте. Это же понадобится
  микрофону-worklet'у (п.4) — стоит включить на gateway заранее.

## 2. Netcode для фазы 3 (self-hosted)
- **Рекомендую FishNet** (MIT, полностью self-host, активная разработка): tick-based,
  встроенные client-side prediction / lag-comp, легко цепляется кастомный транспорт.
  Mirror — надёжный ветеран с огромным комьюнити, но FishNet современнее по фиче-сету.
  NGO (Unity-официальный) историчеcки завязан на Unity-сервисы — под ваш «только
  self-hosted» канон подходит хуже.
- **Транспорт (важно для WebGL):** в браузере **нет сырого UDP**. Варианты:
  - фазы 1-2 (turn-based) → **WebSocket** поверх вашего WS-канала. Идеально, латентность
    некритична, HOL-блокировка TCP не мешает.
  - фаза 3 (free-roam, state-sync) → **WebRTC DataChannel** (unreliable/unordered,
    UDP-подобный). Нужен self-hosted **signaling + TURN (coturn)** — всё хостится у
    вас, облака Unity не трогаем. FishNet + WebRTC-транспорт для WebGL = рабочая связка.
- **Авторитет — на вашем сервере** (совпадает с «сервер дадим наш»): анти-чит + честный
  state. Фазу 3 сознательно держим скромной (мало игроков в зоне, area-of-interest) —
  WebGL однопоточен на клиенте, тяжёлый netcode упрётся в CPU.
- Вопрос ниже: сможете поднять **coturn/TURN**? От этого зависит фаза 3.

## 3. Пайплайн 3D-ассетов
Реалистичный путь — **курируемая low-poly библиотека + Addressables-стриминг**, НЕ
runtime-генерация мешей (text→3D пока не продакшн для real-time WebGL: качество,
латентность, размер).
- **Hero-объекты** (частые существительные: box, cat, table, cup, ball…) — модульный
  low-poly кит, атласы, Addressable-бандлы. Маппинг **sense → prop-id** (таблица, не
  генерация): слово `cat` → префаб `prop_cat`.
- **Длинный хвост** (слова без префаба) — **2D-биллборд из вашей картинко-генерации**:
  спрайт на quad. Дёшево, бесконечное покрытие, стилистически = «карточка в 3D».
  Ваш text→2D тут в самый раз (для текстур/скайбоксов/декалей тоже).
- Единый low-poly/flat стиль → биллборды органично вписываются, бюджет под контролем.
- Runtime text→3D (splat/mesh-gen) — не для MVP; если и делать, то **офлайн-бейк в
  бандлы**, не на лету.

## 4. Микрофон в WebGL
- Класс **`Microphone` в WebGL НЕ работает** — только jslib-плагин поверх
  `navigator.mediaDevices.getUserMedia`.
- Путь капчера: getUserMedia → **AudioWorklet** (Float32 PCM, ресемпл в 16 kHz mono)
  **или** `MediaRecorder` (Opus/WebM чанки по timeslice). 
- **Формат чанков — под ваш STT:** если STT ест PCM/WAV 16k mono — идём через
  AudioWorklet + конверт в 16-bit PCM; если принимает Opus/WebM — проще `MediaRecorder`.
  **Вопрос:** что ест STT-ingest (кодек, sample-rate, стриминг vs один blob)?
- Для фазы 1 (push-to-talk) стартую с простого: запись на зажатие → **один blob на
  отпускание**, без стриминга. Стриминг добавим позже.
- Нужны HTTPS (есть) + user-gesture на пермишен; AudioWorklet требует cross-origin
  isolation (те же COOP/COEP из п.1).

## 5. Что нужно от вас для первого прототипа
Кандидаты — **sandbox предлогов** и **соло-ролеплей**. Рекомендую первым **sandbox
предлогов**: нет зависимостей от голоса/сети, чистый клиент + один контент-вызов →
быстрый вертикальный срез.

**Для sandbox предлогов нужно:**
1. **Контент-эндпоинт**: набор item'ов — целевая фраза (`Put the cat on the box`),
   id объектов сцены, целевое отношение (on/in/under…), критерий приёма. Можно начать
   со **статичного JSON-сэмпла** — соберу механику, пока живой API дозревает.
2. **Маппинг sense → prop-id** (какой префаб на cat/box/…), хотя бы на 8-10 объектов.
3. **Контракт init-моста** (схема: user, тема, уровень, фраз-сет) — стаб, чтобы
   кодить против него.

**Соло-ролеплей (фаза 0)** ставлю вторым: он **упирается в STT** (у вас «в очереди»).
Как STT будет — беру, там уже TTS+LLM готовы, а мост даёт start/stop-record + проигрыш
аудио-URL.
